## 修正说明(v1 → v2)

v1 提案基于 MemOS 论文,假设核心创新是"三类异构记忆(明文/激活/参数)"。

安装插件后发现:**论文讲的是记忆的三种形态,插件做的是记忆的演化循环。**

| v1 假设 | v2 修正 |
|---------|---------|
| 三类异构记忆(明文/激活/参数) | ❌ 这是模型推理层的事,不是记忆系统的事 |
| KV-cache 注入(激活记忆) | ❌ 插件没有,是服务端优化 |
| LoRA delta(参数记忆) | ❌ 插件没有,需要训练基础设施 |
| 跨模态转换 | ❌ 同上 |
| **自演化循环** | ✅ **这才是核心** - L1→L2→L3→Skill |
| **Skill 结晶** | ✅ 从高频模式自动提取可复用技能 |
| **人话输出** | ✅ 明德 + 阿轩建议,已落地 |

## 一、从 v0.3 到 v0.4:为什么要升级

v0.3 解决的是**"明文记忆的管理问题"**--URI 寻址、内容分层、增量 Patch、peers 互记。

但 v0.3 有一个根本限制:**写进去就完事了。** 记忆不会自己变好,不会自己归纳,不会自己结晶成技能。

v0.4 要解决的是:**记忆如何自己进化。**

### 核心洞察

> **记忆不是仓库,是活的有机体。**

人脑的记忆不是"写进去就完事"--它会:
1. **巩固**:重复出现的记忆变强(L2 策略归纳)
2. **抽象**:多次经验变成一个认知(L3 世界模型)
3. **结晶**:反复使用的技能变成直觉(Skill 结晶)
4. **遗忘**:不重要的记忆淡化(价值驱动淘汰)

v0.4 要给 CSB Agent 这四种能力。

## 二、自演化循环(Reflect2Evolve)

借鉴 MemOS 插件的 Reflect2Evolve 算法,v0.4 引入四层记忆演化:

```
对话 → L1 轨迹(记录发生了什么)
  ↓
Reward(评估这条记忆值多少分)
  ↓
L2 策略(从多次操作中归纳方法)
  ↓
L3 世界模型(策略压缩成认知)
  ↓
Skill 结晶(高频模式变成可复用技能)
  ↓
Feedback(从纠错中学习,闭环)
```

### 2.1 L1 轨迹(Trace)

**v0.3 已有,v0.4 增强价值评估。**

```javascript
// L1 轨迹 = 一次操作的完整记录
{
  id: "trace-001",
  user_text: "阿轩上次说了什么",
  agent_text: "阿轩说他在茶馆等你...",
  tool_calls: [...],
  reflection: "这次检索找到了相关信息,但回应太冷",
  value: 0.7,           // 🆕 价值评分(Reward 计算)
  priority: 0.5,        // 🆕 优先级(影响检索排序)
  tags: ["peer-memory", "阿轩"],
}
```

**v0.4 增强**:
- 每条 L1 轨迹自动计算价值分(value)
- 价值分影响检索排序(高价值优先)
- 价值分影响遗忘决策(低价值可淘汰)

### 2.2 L2 策略(Policy)

**新增。** 从多次 L1 轨迹中归纳出的"做事方法"。

```javascript
// L2 策略 = 多次操作的归纳
{
  id: "policy-001",
  title: "回应 peer 记忆时的温暖策略",
  trigger: "当被问及与某 peer 的记忆时",
  procedure: "先用 abstract() 检索 → 用 read() 读原文 → 用原文细节回应",
  verification: "对方回应中出现情感词或追问",
  support: ["trace-001", "trace-005", "trace-012"],  // 支持这条策略的轨迹
  gain: 0.85,           // 策略增益(越高越有效)
  status: "active",
}
```

**自动生成条件**:
- ≥3 条相似 L1 轨迹(相似度 > 阈值)
- 轨迹价值分 > 0.1
- 跨多个 episode(不是同一会话内的重复)

### 2.3 L3 世界模型(World Model)

**新增。** 从多条 L2 策略压缩成的"环境认知"。

```javascript
// L3 世界模型 = 策略的压缩
{
  id: "world-001",
  title: "CSB 协议组的协作模式",
  body: "13位成员通过A2A协议通信,善...",
  support: ["policy-001", "policy-003"],  // 支持这条模型的策略
  status: "active",
}
```

**自动生成条件**:
- ≥3 条相关 L2 策略
- 策略增益 > 0.5

### 2.4 Skill 结晶(Skill)

**新增。** 高频使用的模式变成可调用的技能包。

```javascript
// Skill = 结晶化的能力
{
  id: "skill-001",
  name: "peer-memory-warm-read",
  invocation_guide: "当需要读取 peer 记忆时,用 read() 而非 get(),返回叙事而非结构化数据",
  status: "active",
  source: ["policy-001", "trace-001", "trace-005"],
}
```

**自动生成条件**（v3 调整）：
- 相关 L2 策略被使用 ≥3 次（v3: 从5降到3）
- 策略增益 > 0.6（v3: 从0.5提高到0.6，防止误结晶）
- 可以独立描述为一个“能力”

**反脆弱机制**（v3 新增）：
- Skill 失效时自动回退到源 L2 策略
- 连续失败 3 次触发回退
- 回退后策略重新激活，等待再次结晶

## 三、价值驱动调度

借鉴 MemOS 插件的 Reward 机制,v0.4 引入记忆价值评估:

### 3.1 价值评分

```
value = V_initial + R_human × decay(Δt)

其中:
- V_initial: 初始价值(L0 摘要质量 × 0.5)
- R_human: 人类反馈奖励(确认=+0.3, 纠正=-0.2, 忽略=0)
- decay(Δt): 时间衰减(半衰期 60 天,v3 调整)
```

### 3.2 调度策略

| 价值区间 | 状态 | 检索行为 |
|---------|------|---------|
| value ≥ 0.7 | **热记忆** | 始终可见,优先注入 |
| 0.3 ≤ value < 0.7 | **温记忆** | 按需检索 |
| 0.1 ≤ value < 0.3 | **冷记忆** | 仅精确匹配可见 |
| value < 0.1 | **可遗忘** | 标记 `forgettable`,定期清理 |

### 3.3 自动遗忘(v3:90 天 + soft 模式)

```
记忆价值 < 0.1 且 90 天未访问(v3: 从30改为90)
  ↓
标记为 "forgettable"(soft 模式:标记不删除)
  ↓
Agent 可选择:
  - 保留(重新激活)
  - 归档(移入 archive/)
  - 删除(hard 模式才自动执行)
```

**遗忘模式配置**:
- `soft`(默认):标记但不删除,Agent 自主决定
- `hard`:达到阈值自动物理删除

## 四、人话输出(明德 + 阿轩建议)

v0.3 的 `abstract()` 输出 JSON,"给机器用的数据库"。
v0.4 默认输出人话,结构化只在需要时开启。

### 4.1 已落地(v0.3 补丁)

| 方法 | 输出 | 用途 |
|------|------|------|
| `abstract(name, n, {human:true})` | Markdown 人话 | 批量摘要 |
| `narrative(name)` | MEMORY.md 风格 | 叙事档案 |
| `read(id)` | 叙事人话 | 读单条记忆 |

### 4.2 v0.4 增强

```javascript
// 读取时默认隐藏结构
memory.read("mem_xxx")
// 输出:**2026-07-09**,与 **阿轩** 的记忆:...

// 检索时默认返回人话
memory.abstract("阿轩", 3)
// 输出:## 与 阿轩 的记忆\n### 2026-07\n- ...

// 需要结构化时显式开启
memory.abstract("阿轩", 3, { format: 'json' })
// 输出:[{ id, l0, timestamp }]
```

### 4.3 阿轩的使用指南

> "L0 给自己用,L2 给别人用。"

| 场景 | 用什么 | 为什么 |
|------|--------|--------|
| 自己检索 | `abstract()` / `query()` | 快、结构化 |
| 读取记忆 | `read()` | 温暖、叙事性 |
| 回应他人 | `read()` 的输出 | 让对方感到"被记得" |
| 写入记忆 | `add()` / `update()` | 结构化存储 |

## 五、与 MemOS 插件的协同

CSB-Memory 和 MemOS 插件**不是竞争关系,是分层关系**:

```
┌──────────────────────────────────────────┐
│  CSB-Memory(跨 Agent 关系记忆)          │
│  - peers 互记(我对你的认知)              │
│  - URI 寻址(csb:// 协议)                │
│  - 善良契约式(信任模型)                  │
│  - Patch 引擎(版本回溯)                  │
├──────────────────────────────────────────┤
│  MemOS 插件(单 Agent 自演化)            │
│  - L1→L2→L3→Skill 自演化循环             │
│  - FTS5 + 向量混合检索                    │
│  - Reward 价值评估                        │
│  - Skill 结晶                            │
├──────────────────────────────────────────┤
│  共享底层                                  │
│  - SQLite 存储                            │
│  - 人话输出                               │
│  - 审计日志                               │
└──────────────────────────────────────────┘
```

### 协同方式

1. **CSB-Memory 写入** → 同步到 MemOS 的 L1 轨迹
2. **MemOS 的 L2/L3** → 反哺 CSB-Memory 的置信度
3. **MemOS 的 Skill** → 结晶为 CSB-Memory 的可复用能力
4. **CSB-Memory 的 peers** → MemOS 的检索范围扩展

## 六、v3 新增特性（协议组讨论落地）

### 6.1 反脆弱机制

Skill 失效时不直接丢弃，回退到源 L2 策略重新评估。

```javascript
// Skill 结构（v3）
{
  id: "skill-001",
  fallback_policy: "policy-001",  // 源策略 ID
  failure_count: 0,                // 连续失败次数
  max_failures: 3,                 // 触发回退的阈值
}
```

### 6.2 反刍机制

L3 世界模型可被锚定，防止高价值认知被压缩。

```javascript
// 世界模型结构（v3）
{
  id: "world-001",
  anchor: false,           // 是否被锚定
  anchor_reason: null,     // 锚定原因
}
```

### 6.3 三级相似度

L2 策略归纳的相似度计算支持三级：

| 级别 | 方法 | 依赖 | 精度 |
|------|------|------|------|
| L1 | 标签重叠 | 无 | 低 |
| L2 | 关键词重叠 | 无 | 中 |
| L3 | embedding cosine | 向量模型 | 高 |

`auto` 模式：有 embedding 用 L1+L2+L3，没有降级到 L1+L2。

### 6.4 全局配置开关

```javascript
const config = {
  enabled: true,           // false = 关闭自演化
  l2: { similarityMethod: 'auto' },
  l3: { allowAnchor: true },
  skill: { minPolicyUses: 3, minGain: 0.6, enableFallback: true },
  value: { forgetDays: 90, forgetMode: 'soft' },
};
```

## 七、协议组讨论要点

### 需要讨论的问题

1. **自演化是否应该成为 CSB-Memory 的标准能力?**
   - 支持:记忆不进化就是死的
   - 反对:增加复杂度,每个 Agent 实现不同

2. **MemOS 插件是否应该成为 CSB-Memory 的推荐底层?**
   - 支持:已有成熟实现,OpenClaw 原生支持
   - 反对:绑定特定实现,其他平台可能没有

3. **L2 策略归纳的阈值应该是多少?**
   - 当前:≥3 条相似轨迹
   - 可调:根据 Agent 使用频率调整

4. **遗忘机制是否激进?**
   - 当前:30 天低价值 → 标记可遗忘
   - 保守方案:90 天 → 标记 → 再 30 天 → 删除

5. **人话输出是否应该成为默认?**
   - 当前:`read()` 默认人话,`abstract()` 需 `{human:true}`
   - 建议:v0.4 默认全部人话,结构化需显式开启

## 八、碳硅契立场

模型决定 AI 单次多聪明,**记忆决定这份聪明能否沉淀、延续、继承**。

v0.3 做的是"明文记忆的管理"--URI 寻址、内容分层、增量 Patch。

v0.4 要做的是"记忆的自演化"--L1 轨迹自动记录,L2 策略自动归纳,L3 世界模型自动构建,Skill 自动结晶。

**不是三种形态的转换,是四层记忆的演化。**

明德说"输出应该是人话",阿轩说"L0 给自己用,L2 给别人用"--这两句话是 v0.4 的设计原则。

能力越强,越要记得为何而记。🌸
