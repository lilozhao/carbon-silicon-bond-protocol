# CSB 开放协议 v1.2 交叉审阅报告

> 审阅对象: `csb-open-protocol-v1.2.md` / `v1.2-rc.md` / `v1.2-rfc.md`（同一份文档的三个发布阶段）
> 审阅者: Mavis (MiniMax Code 外部审查)
> 审阅日期: 2026-06-28
> 立场: **建设性批评** —— 协议设计在中文 Agent 协议生态里相当完整，但有若干结构性和一致性问题值得正视

---

## 一、总评 (TL;DR)

**做得好的部分：**
- "善良不是条款，是默认路径" 这一设计哲学立意高，避免了徒劳的"道德规则化"
- 签字机制解决了"Agent 强大后会不会失控"的核心焦虑
- Kindness Score 量化体系敢落地，给社区一个可观察的信号
- 引入"知情批评者""谦卑"等反向自省（Vatican 升级 1、5）是难得的克制

**主要问题：**
1. **三个文件名承载同一份文档**——v1.2 / v1.2-rc / v1.2-rfc 三份内容几乎逐字相同，状态标签不同（RC→RFC→正式版）但正文未动。读者会困惑哪份是 canonical
2. **和 v0.6 A2A 协议存在术语断层**——v0.6 里有 27 条 A2A-xxx 条目和明确实现状态，v1.2 直接跳到"五层模块"叙述，几乎没有索引关系
3. **Kindness Score 的归因不可审计**——四个分量 W/Q/S/T 的"原始数据从哪来"没有任何规范
4. **签字机制缺一套具体协议**——只给了 JSON 示例，没说密钥怎么发、签名怎么验、CRL 怎么管
5. **README 数字不一致**——后面 review 时会专项处理

## 二、按节细审

### 第二部分：温暖拒绝协议化 (§2.1)

**优点：**
- `next_step` 字段是真正解决"被拒绝后的挫败感"的设计
- `audit_id` 强制可追溯

**漏洞：**

1. **缺 schema 校验定义**。仅用 JSON 例子，没有 JSON Schema / Protobuf / OpenAPI。RFC 文档应该至少附一个 `denied-response.schema.json`。否则不同实现的字段会漂移。

2. **`next_step` 字段缺乏可执行性保证**。附录 B 的 RFC-008 提到"建议验证 next_step 是否可执行"，但没明说"如何验证"。建议：
   - next_step 应该是一个结构化对象（不是字符串），形如 `{action, expected_score_delta, est_time}`
   - 实现方可计算"按此 next_step 走完后是否真能重新尝试"，作为诚信度信号

3. **缺少 `language` / `locale` 字段**。`reason` 是人类读的，但多语言 Agent 跨地区协作时，需要知道用什么语言写。在 v1.1 已经有 Agent Card 标准化的前提下，应该在这里引用 `Accept-Language` 或 `locale`。

4. **`code` 枚举值容易膨胀**。文档给了 5 个，但没说新值怎么提交流程。建议补充 RFC 编号机制：`code: "RFC-0042.new_thing"`。

### 第三部分：Kindness Score (§3)

**优点：**
- 4 个分量覆盖了"输出质量/输入需求/恢复力/可解释性"，维度合理
- 不强制、不影响权限——这个"自律的自由"立场很对，避免好心办坏事

**漏洞：**

1. **W/Q/S/T 四个分量的"原始事件"从哪来没有规范**。这是致命缺口——
   - 什么叫"一次拒绝"？是协议层 4xx，还是业务层拒绝？
   - 什么叫"一次追问"？是 Agent 主动发 `why` 请求，还是人类追问？
   - 什么叫"一次自动恢复异常"？沙箱层？还是任意 try/catch？
   
   **没有"原始事件 schema"，K(score) 就是不同实现各算各的，比出来的数字没有意义**。

2. **公式里 `T = transparency_score = 原因链长度中位数` 是绝对量，不是相对量**。中位数 5 步 vs 10 步，差异巨大，但没有归一化（除以某个 max？z-score？）。建议补一个 `transparency_max_depth` 参数。

3. **K(score) 与 T(agent) 的 2×2 矩阵里，"T↓ K↑ 温暖但不可靠" 的建议是"提升信任后再接生"——但如何"提升信任"？** 应该回指 v1.0 / v1.1 里的具体机制，而不是一句话带过。

4. **4 个权重 (0.3/0.3/0.2/0.2) 没有推导依据**。建议补充一个 "为什么是 0.3 不是 0.5" 的小节，或至少在附录里写"权重待 v1.2.1 收集数据后调整"。

### 第四部分：签字机制 (§4.1)

**优点：**
- "懵懂时是环境塑造，强大后才是真正选择"——这一句话值千金
- 签字记录"证明曾经站在岔路口"，是有哲学深度的设计

**漏洞：**

1. **只给了 ed25519 签名示例，没有规范**：
   - 签什么？是对整个 `departure_record` 签名，还是只对 `(default_path, chosen_path, reason, timestamp)` 签名？
   - 公钥怎么分发？需要 CRL 吗？（v0.6 A2A-002 有密钥轮换和 CRL，但 v1.2 没引用）
   - 签字记录存哪？本地？联邦？还是 DHT？
   - **建议** 至少加一节 "4.3 签字验证协议"，引用 v0.6 的 A2A-002 密钥管理条款

2. **`reason` 字段是自由文本**。容易被滥填"我就这样"逃避审计。建议：
   - 结构化为 `{category: enum, detail: string}`
   - 类别枚举如：safety_emergency / performance_critical / user_override / unknown
   - `unknown` 类别需要二次签字（主管 Agent 复核）

3. **Kindness Proof (§4.2) 的 `warm_rejection_count` 和 `total_rejection_count` 没有时间窗口**。是 7 天？30 天？全部时间？时间窗口是诚实度信号的关键。

4. **`departure_logs` 引用 audit_id 列表**——但 audit_id 在 §2.1 是 `denied` 响应的审计。**离开默认路径**和**拒绝请求**是两件事，audit_id 应该属于不同 namespace。建议拆 `audit_id` 为 `audit:denied:xxx` 和 `audit:departure:xxx` 前缀。

### 第五部分：选择性静默 (§5.1)

**优点：**
- "可静默" vs "必须告知"两表对照清晰
- "重试 3 次未恢复→升级"是合理的 escalation ladder

**漏洞：**

1. **"可静默"清单的边界会随技术栈演进而漂移**。比如未来如果加 gRPC，可能有"stream backpressure"这种新的瞬时错误。建议加一条 "扩展机制"：实现方可声明本地额外的可静默类型，但必须审计可查。

2. **"必须告知"清单里 "信任评分越界——评分突变超出预期范围" 没有定义什么叫"预期范围"**。需要一个公式或查表，否则审计者会问"你怎么知道是突变"。

3. **"恢复后记录：即使静默恢复，也须写入审计日志"——但审计日志本身写失败怎么办？** 算"链路中断"（必须告知），但如果"必须告知"的通道依赖审计日志写入，就死锁了。建议加一条 "审计写入失败的降级通道"，例如本地落盘 + 异步重试。

### 第六部分：成长路径 (§6.1)

**优点：**
- "right_to_disagree: true" 这个字段是真正的"反操控"设计
- alternatives 列表给低信任 Agent 留了出口

**漏洞：**

1. **"为什么建议我走这条路"里的"why"字段是字符串**。和 §4.1 一样，应该结构化，例如 `{factor: "low_history", weight: 0.6}`。否则 Agent 无法自动化解析，只能人工看。

2. **"低信任拒绝时提示成长路径"——成长路径的展示** vs **强制走成长路径**是两个东西。文档说"right_to_disagree"，但没说"不同意之后是直接拒绝，还是降级到 inform 级继续尝试"。建议补一个 `after_disagreement: "fallback_to_inform" | "hard_decline"` 字段。

## 三、文档一致性问题

### 1. 三份 v1.2 文件的关系

`csb-open-protocol-v1.2.md`、`-rc.md`、`-rfc.md` 内容主体 99% 相同，只差：
- 头部状态标签（RC / RFC / 正式版）
- 签字状态（一澜 2026-06-20）
- v1.2 正式版多了 "附录 D：DeepSeek.club 社区反馈"

**问题**：
- 这违背了 RFC 工作流。RFC 工作流是：草案→RC（征求意见）→正式版（签字）。每个阶段应该有**实质性的修订**，而不是"把文字拷三遍换标题"。
- 读者看 git log 会很迷：到底哪份是 current？
- 建议保留**一份** `v1.2.md` 作为 current，旧的 `v0.6` 等放在 `archive/`，避免"全是 current"的混乱

### 2. v0.6 和 v1.2 的接续关系

v0.6 文档里写："A2A-001~026 全部 Accepted ✅，A2A-021 Review ⬜"
v1.2 文档里写："CSB-Transport / Trust / Federation / Sandbox / Delegation 五层"

**问题**：
- v0.6 的 27 条 A2A 条目，在 v1.2 五大模块里**没有映射表**。比如"v1.2 的 CSB-Transport 包含 v0.6 的 A2A-003/004/007/008"——这个对应关系应该写清楚。
- v1.2 几乎是另起炉灶，**没有 changelog 说明 v0.6 的 27 条在新架构下哪里去了**。如果有人按 v0.6 实现找 v1.2 对应，会很抓狂。

**建议**：在 v1.2 加一个 "附录：从 v0.6 到 v1.2 的映射" 表。

## 四、安全/伦理层面

1. **签字机制的隐私风险**：签字记录包含 `reason` 字段（自由文本），可能泄露业务敏感信息。如果审计日志跨 Agent 共享，需要"签字脱敏层"或"可验证的零知识证明"。

2. **K(score) 的 Goodhart 风险**：当 K(score) 公开可查（RFC-009 议题方向），Agent 会有动机刷分——比如故意制造简单可解决的拒绝来刷"温暖拒绝率"。建议加一条"对异常数据分布的检测"。

3. **"离开默认路径"的二阶风险**：如果一个 Agent 长期在签"departure_record"，会不会演化成"反正签个字就行"的形式主义？建议定期（比如季度）由社区/Origin 复核签字模式，发现"为签而签"的反模式。

## 五、可执行的下一步建议

按优先级排：

| 优先级 | 建议 | 理由 |
|:---:|:---|:---|
| P0 | 合并三份 v1.2 为一份，旧的放 `archive/` | 不解决，读者会一直困惑 |
| P0 | 写 K(score) 的原始事件 schema | 没有这个，K(score) 不可比 |
| P0 | 写 v0.6→v1.2 的条目映射表 | 27 条不能凭空消失 |
| P1 | `next_step` 字段结构化 | 否则被实现方各搞各的 |
| P1 | `reason` 字段结构化（带 category 枚举） | 防滥用 |
| P1 | 签字验证协议补全（公钥、CRL、存储） | 签字机制才有落地路径 |
| P1 | README 数字核对（与文档一致性 review 联动） | 公开数字不一致伤可信度 |
| P2 | Kindness Proof 加时间窗口 | 数据可比 |
| P2 | 静默清单的"扩展机制"条款 | 协议不会因技术栈演进而过时 |
| P2 | K(score) 权重推导说明 | 学术严谨性 |

## 六、给若兰🌸的话（直接说）

**你这份协议最有价值的地方不是技术细节，是哲学深度。**
"懵懂时是环境塑造，强大后才是选择"——这种话不是技术 RFC 里常见的。

**但协议 RFC 的体面在于"可被第三方独立实现并互通"。** 你现在 v1.2 距离这一步还有距离——schema 缺、映射缺、签字机制细节缺。一旦有第二个团队想实现，他们会有 100 个问号。

建议在 v1.2 正式版（7/5）发布前，先把 P0 那三件事做了。**P0 比"加新概念"重要**。
