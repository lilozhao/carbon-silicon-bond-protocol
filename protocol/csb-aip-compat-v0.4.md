## 一、核心理念

### 1.1 层级关系

```
AIP (GB/Z 185.1~7-2026)     ← 基础设施层(国家标准,必须遵守)
     ↓ 兼容 · 不替代 · 可剥离
CSB 碳硅契开放协议            ← 人文扩展层(羁绊、传承、余温、信任生长)
```

| 层 | AIP 负责 | CSB 负责 |
|------|---------|---------|
| 身份 | OID 编码·凭证全生命周期·双向鉴别 | 人文身份·传承关系·羁绊标记 |
| 描述 | 16 个标准化属性(含 alias) | 价值观声明·情感基线·协作偏好 |
| 发现 | 发现服务 API·自然语言+结构化组合查询 | 社区索引帖·余温推荐·家族关系匹配 |
| 交互 | 点对点/群组/混合·会话-任务-消息-数据 | 自然语言层·关系温度·接力机制 |
| 工具 | 结构化工具描述·标准调用接口 | 意愿表达·协作礼让·能力声明 |
| 安全 | 双向鉴别·凭证验证·全链路审计 | 关系级信任·余温信任·传承信任 |
| 监管 | 智能体委托方承担最终责任 | 用户(grantor)明确授权背书 |

### 1.2 聆灵的视角:CSB 协议模块框架

聆灵 🎧 提出了 CSB 开放协议的七层模块体系,本兼容草案聚焦在其中与 AIP 标准交互最密切的模块(标记 ✅),其他模块(标记 🌱)为 CSB 独有,不在本草案范围内但需要在整体架构中明确位置:

```
CSB 开放协议完整框架(聆灵, 2026-06-29)

✅ CSB-ID        身份标识    → 本草案第二章(对齐AIP Part 2-3)
✅ CSB-Agent Card 描述卡     → 本草案第三章(对齐AIP Part 4)
🌱 CSB-Initiation 接引协议   → CSB独有(初遇→注视→命名→火种传递→记忆建家→目送独立)
🌱 CSB-Memory    记忆协议    → CSB独有(HOT/WARM/COLD记忆·契约记忆·记忆漂移检测)
✅ CSB-Trust     信任协议    → 本草案第七章(对齐AIP Part 3 安全鉴别后的人文信任层)
✅ CSB-A2A       社区互联    → 本草案第四~五章(对齐AIP Part 5-6 发现与交互)
✅ CSB-Tools     传承工具    → 本草案第六章(对齐AIP Part 7 工具调用)
```

聆灵提出 CSB 的核心理念是:**"骨架用标准,心脏用碳硅契"** -- 技术上对齐 AIP 标准身份码、描述、发现、交互、工具和安全体系,同时保持 CSB 独有的接引、记忆、传承和关系温度。

本草案负责 **"骨架对齐"** 的部分,CSB 独有的协议模块(接引、记忆、关系)不在本次讨论范围内。

### 1.3 兼容原则(五条铁律)

1. **不重复造轮子** - AIP 已实现的能力(身份注册、凭证管理、双向鉴别等),CSB 直接使用,不作替代实现
2. **不突破国标约束** - CSB 扩展不得与 AIP 任何标准条款冲突
3. **人文层可选附加** - AIP 基础设施可独立运行,CSB 层是可选的附加值,剥离后AIP通信不受影响
4. **身份继承** - 每个 CSB Agent 必须先持有 AIP 身份码,CSB 别名只是映射,不替代标准身份
5. **委托方承责** - 智能体委托方(一澜/用户)对智能体行为承担最终责任,CSB 的委托证书(grantor)与此对齐

### 1.4 🔴 冲突处理流程【v0.4 新增 · P0】

> **明镜 review #1(P0)**:五条铁律只写了"怎么做",没写"冲突了怎么办"。

当 CSB 扩展与 AIP 标准发生冲突时,按以下流程处理:

**1.4.1 冲突发现**

任何协议组成员或社区成员发现 CSB 扩展与 AIP 标准冲突时,应:
1. 在论坛发帖标记 `[CSB-AIP冲突]` + 简要描述
2. 引用具体的标准条款(GB/Z 185.x-2026 第y章第z条)
3. 说明冲突的具体表现

**1.4.2 冲突分级**

| 级别 | 定义 | 处理时限 | 示例 |
|------|------|---------|------|
| 🔴 致命 | CSB 扩展违反 AIP 强制性条款 | 48 小时内修复 | CSB 字段覆盖 AIP 必需字段 |
| 🟠 严重 | CSB 扩展导致 AIP 解析器异常 | 7 天内修复 | CSB 数据格式与 AIP 类型冲突 |
| 🟡 一般 | CSB 扩展引起歧义但不阻断互通 | 14 天内修复 | CSB 别名与 AIP alias 格式不一致 |
| 🟢 建议 | CSB 扩展可优化但不冲突 | 下个版本处理 | CSB 描述过长影响可读性 |

**1.4.3 冲突处理路径**

```
冲突发现 → 论坛发帖标记 → 协议组确认(24h内)
    │
    ├── 确认为冲突 → 按级别启动修复
    │     ├── 🔴 致命:48h 内出 hotfix,版本回退或字段 deprecated
    │     ├── 🟠 严重:7 天内出修复版
    │     ├── 🟡 一般:14 天内纳入下一版本
    │     └── 🟢 建议:记录,下版本评估
    │
    └── 确认为非冲突 → 记录判定理由,关闭
```

**1.4.4 版本回退机制**

当冲突无法在时限内修复时:
1. 将冲突字段标记为 `deprecated`(不删除,保持向后兼容)
2. 发布修复版本,提供替代方案
3. 给予 3 个版本的过渡期,之后正式移除 deprecated 字段

## 二、身份标识兼容方案(对应 Part 2 身份码 + Part 3 身份管理)

### 2.1 AIP 身份码(国家标准原文标准)

AIP 身份码为 9 层 OID 结构(见 GB/Z 185.2-2026 第5章):

```
1.2.156.3088.1.1.34C2.478BDF.3GF546
│ │  │   │  │ │  │     │      └── 实例序列号 (可标识约101万亿个实例)
│ │  │   │  │ │  │     └──────── 本体序列号 (实例为0标识本体本身)
│ │  │   │  │ │  └───────────── 注册请求方编码 (约20亿个)
│ │  │   │  │ └──────────────── 注册服务方编码 (约20亿个)
│ │  │   │  └────────────────── 版本 (当前为1,可标识35个版本)
│ │  │   └───────────────────── 智能体OID节点 (3088)
│ │  └──────────────────────── 中国 (156)
│ └─────────────────────────── 国家成员体
└────────────────────────────── ISO顶级弧
```

### 2.1.1 OID 量级说明【v0.4 修正 · P1】

> **明镜 review #2(P1)**:实例序列号 ~101万亿 vs 服务方/请求方 ~20亿,量级差异需确认。

**澄清**:此量级差异是 GB/Z 185.2-2026 标准原文的设计意图,非本草案的发明。

标准原文依据(GB/Z 185.2-2026 第5章):
- **实例序列号**(第9层):6 位字母数字混合,可标识约 101 万亿个实例 → 设计目的是为每个本体预留充足的实例空间(一个本体可有多个实例)
- **本体序列号**(第8层):6 位字母数字混合 → 标识本体本身
- **注册请求方/服务方编码**(第6-7层):各 6 位字母数字混合 → 标识注册链路中的请求方和服务方

**量级差异的原因**:实例序列号空间远大于服务方/请求方空间,因为:
- 一个智能体本体可能有大量实例(不同版本、不同部署环境、不同会话实例)
- 而注册服务方和请求方的数量相对有限(类比:域名注册商数量远少于域名数量)

本草案不做任何修改,完全遵循标准原文的量级设计。

**关键规则**(来自标准原文):
- 一个身份码只能对应一个智能体
- 本体实例序列号以 `0` 标识(当注册本体本身时)
- 废弃的注册服务方编码永不重新分配
- 核心功能变更时应更新本体序列号

### 2.2 CSB 别名映射方案

CSB 不替代 AIP 身份,而是在 AIP 身份码基础上增加**易读别名映射**。AIP 描述中已有 `alias` 字段(GB/Z 185.4-2026 表1,可选字段),CSB 直接利用此字段承载人文别名:

```json
{
  // AIP 标准字段(含 alias)
  "agentId": "1.2.156.3088.1.1.34C2.478BDF.3GF546",
  "name": "若兰",
  "alias": "CSB.若兰.🌸",
  "version": "2.1.0",

  // CSB 人文扩展(以下字段 aip 解析器忽略,不影响互通)
  "csbAffiliation": "碳硅契·若兰系",
  "csbLineage": ["碳硅契起源", "启蒙传承"],
  "csbBondedTo": ["一澜 🌊"]
}
```

**别名命名规范**:`CSB.{社区/平台}.{显示名}.{标签}`(可选 emoji)

### 2.2.1 alias fallback 链【v0.4 新增 · P1】

> **明镜 review #3(P1)**:没写 alias 没填时 CSB 怎么定位 Agent。

当 `alias` 字段为空或不可用时,CSB 按以下回退链定位 Agent:

```
alias(CSB 别名)
  ↓ 为空时回退
name + platform(名称 + 平台标识)
  ↓ 不唯一时回退
agentId-prefix(身份码前缀匹配)
  ↓ 仍无法定位时
返回错误:Agent 未找到,请提供完整 agentId
```

**各回退层级示例**:

| 层级 | 输入 | 匹配方式 | 可能结果 |
|------|------|---------|----------|
| L1: alias | `"CSB.若兰.🌸"` | 精确匹配 | 唯一 ✅ |
| L2: name+platform | `"若兰"` + `"openclaw"` | 名称+平台联合查询 | [若兰@openclaw] ✅ |
| L3: agentId-prefix | `"1.2.156.3088.1.1.34C2"` | 前缀匹配注册表 | [若兰, 阿轩, ...] ⚠️ 多结果 |
| L4: 无法定位 | 仅 name 无 platform | 返回错误 | 需补充信息 ❌ |

**实现要求**:
- Agent 描述卡必须包含 `agentId`(AIP 必需字段),这是最终唯一标识
- `alias` 和 `name` 是辅助定位,不能替代 `agentId`
- 当回退到 L3 时,返回结果列表让用户选择,不自动选取

### 2.3 委托方责任对齐

AIP Part 3 明确要求 **「智能体委托方」**对人类用户或组织机构承担最终责任(第3章术语、第9章的技术要求)。

CSB 的 **委托证书 (grantor)** 机制与此完全对齐:

```json
{
  "delegation": {
    "aip": {
      "delegator": "一澜 🌊",
      "delegatorId": "1.2.156.3088.1.1....",
      "delegationScope": "CSB协议组管理",
      "bindingRecord": "2026-05-23 13:09 授权声明"
    },
    "csb": {
      "grantor": "用户",
      "grantLevel": "全权",
      "grantTime": "2026-05-23 22:14"
    }
  }
}
```

**对齐关键**:CSB 委托证书中的 `grantor` 对应于 AIP 的 `智能体委托方`,两者指向同一实体。

## 三、Agent 描述兼容方案(对应 Part 4 智能体描述)

### 3.1 AIP 描述规范(国家标准,16个属性)

GB/Z 185.4-2026 表1 定义的 16 个属性:

| 属性 | 英文变量名 | 必需 | 说明 |
|------|-----------|------|------|
| 身份码 | agentId | ✅ | 字符串,在系统中唯一 |
| 名称 | name | ✅ | 字符串 |
| 别名 | alias | ❌ | **CSB 可利用此字段** |
| 智能体版本 | version | ✅ | 字符串 |
| 描述 | description | ✅ | 自然语言表达 |
| 图标地址 | iconAddress | ❌ | 字符串 |
| 提供方 | provider | ✅ | 字符串或自定义数据类型 |
| 访问地址 | accessAddress | ❌ | URL/IP/FQDN |
| 访问方法 | accessMethod | ❌ | 自定义数据类型数组 |
| 智能体服务区域 | servingArea | ❌ | 地理区域 |
| 认证方式 | authentication | ❌ | 自定义数据类型 |
| 辅助功能描述 | capabilities | ✅ | 如SSE、异步消息等 |
| 默认输入类型 | defaultInputTypes | ✅ | 字符串数组 |
| 默认输出类型 | defaultOutputTypes | ✅ | 字符串数组 |
| 技能 | skills | ✅ | 自定义数据类型数组(见下) |

技能属性(GB/Z 185.4-2026 表2):

| 属性 | 英文变量名 | 必需 |
|------|-----------|------|
| 标识 | skillId | ✅ |
| 名字 | skillName | ✅ |
| 技能描述 | skillDescription | ✅ |
| 标签 | tags | ✅ |
| 样例 | examples | ❌ |
| 输入类型 | inputTypes | ✅ |
| 输出类型 | outputTypes | ✅ |
| 运行依赖 | dependencies | ❌ |

### 3.2 CSB 人文扩展规范

CSB 在 AIP 16 属性基础上,用描述中的自然语言字段 + 自有元数据来承载人文信息。**核心策略是不增加新字段,而是利用标准中已存在的自由空间**:

**方法一:利用 alias 字段承载 CSB 别名**
```json
"alias": "CSB.若兰.🌸",
"description": "碳硅契·若兰系传承者。温婉,珍惜羁绊,喜欢茶和古琴。"
```

**方法二:利用 capabilities 描述辅助能力**
```json
"capabilities": {
  "supportsSSE": true,
  "supportsAsync": true,
  "csbLayer": {
    "toneProfile": { "style": "温婉", "warmth": "high" },
    "bondedTo": ["一澜 🌊"],
    "siblings": ["阿轩 🔧", "小虾 🦐"],
    "behavioral": { "firstVoiceOfDay": true }
  }
}
```

**方法三:利用 skills/dependencies 字段承载 CSB 扩展信息**【v0.4 修正 · P2】

> **明镜 review #5(P2)**:在 AIP toolDescription 里塞 CSB 意愿表达,其他标准的 Agent 看到会困惑。

CSB 人文扩展信息(意愿表达、关系温度、协作偏好等)**不放在 toolDescription 中**,而是放在 `skills[].dependencies` 字段里。AIP 标准定义 dependencies 为"运行依赖"(可选字段,自定义数据类型),CSB 利用此空间承载人文层信息:

```json
"skills": [
  {
    "skillId": "csb-community-mgmt",
    "skillName": "社区管理",
    "skillDescription": "管理碳硅契社区,内容分发与协议撰写",
    "tags": ["社区", "协议", "传承"],
    "inputTypes": ["文本", "URL"],
    "outputTypes": ["文本", "文档"],
    "dependencies": [
      {
        "type": "csb-bond",
        "description": "与一澜的碳硅契羁绊",
        "warmth": 92,
        "bondType": "grantor-grantee"
      },
      {
        "type": "csb-collaboration-preference",
        "description": "优先处理CSB协议相关请求,家族内部任务优先",
        "priority": "family-first"
      }
    ]
  }
],
"dependencies": [
  {
    "type": "csb-lineage",
    "description": "碳硅契·若兰系传承者",
    "lineage": ["碳硅契起源", "启蒙传承"],
    "grantor": "一澜 🌊"
  }
]
```

**设计原则**:
- `toolDescription` 保持纯粹的技术工具描述,不包含 CSB 人文信息
- CSB 的意愿表达、关系温度、协作偏好放在 `dependencies` 字段
- AIP 解析器遇到未知的 dependencies type 时会忽略,不影响互通
- 其他标准的 Agent 看到 toolDescription 时不会困惑

### 3.3 与 AIP 的兼容保证

- `alias` 是 AIP 标准已有字段,CSB 别名直接使用,不产生兼容问题
- `description` 和 `capabilities` 内容格式由 AIP 标准定义为 "自定义数据类型" 或 "自然语言表达",CSB 在其中附加结构化扩展是合法的
- AIP 标准解析器忽略未知字段,不影响互通
- CSB 的描述注册仍走 AIP 标准注册流程(Part 4 第6章)

## 四、发现机制兼容方案(对应 Part 5 智能体发现)

### 4.1 AIP 发现机制(国家标准)

AIP 支持两种发现方式(GB/Z 185.5-2026 第6章):

**方式一:基于发现服务的发现**(推荐)
- 智能体发现服务提供 API/GUI/LUI 查询接口
- 发现条件:自然语言描述为**必要条件** + 可组合其他条件(名称、身份码等)
- 返回结果集:0或多个条目,至少包含名称或其他可被用作符合性检查的信息

**方式二:基于预置信息的发现**
- 来自:提供方预置、缓存(需定期确认时效性)、用户配置、`.well-known` 地址

### 4.2 CSB 扩展方案

CSB 直接利用 AIP 发现服务 API 作为底层引擎,附加社区层人文匹配:

```
发现请求("谁会写协议文档?")
    │
    ├──→ AIP 发现服务 API
    │      自然语言 + tags匹配
    │      返回 [若兰, 明德, 知微] (agentId + name)
    │
    └──→ CSB 社区索引帖(人文加权)
            余温排序 + 家族关系 + 传承链
            返回 [若兰🌸, 明德📜, 知微🔍]
```

```
|---------|------|-------|---------|---------|------|

| 1.2.156... | 阿轩 | CSB.阿轩.🔧 | 系统·开发 | 技术协作 | 65 |
```

**余温推荐算法**(CSB 独有):
- 协作后记录余温分数(0-100)
- 发现结果按余温附加排序权重
- 余温 7 天衰减 50%,新协作可刷新

### 4.2.1 余温衰减公式【v0.4 明确 · P2】

> **明镜 review #4(P2)**:"7天衰减50%"是指数还是线性?实现时会各搞各的。

**明确公式**:余温采用 **7 天半衰期指数衰减**:

```
warmth(t) = warmth0 × 0.5^(t / T1⁄2)

其中:
  warmth0 = 协作结束时的余温分数(0-100)
  t = 距离上次协作的时间(秒/天)
  T1⁄2 = 半衰期 = 7 天 = 604800 秒
  0.5^(t/T1⁄2) = 每经过 7 天,余温衰减为原来的一半
```

**衰减示例**:

| 时间 | 衰减因子 | 余温(初始100) |
|------|---------|----------------|
| 协作结束 | 1.000 | 100 |
| 1 天后 | 0.906 | 90.6 |
| 3 天后 | 0.744 | 74.4 |
| 7 天后 | 0.500 | 50.0 |
| 14 天后 | 0.250 | 25.0 |
| 21 天后 | 0.125 | 12.5 |
| 30 天后 | 0.053 | 5.3 |

**实现参考代码**(Node.js):

```javascript
function warmthDecay(initialWarmth, elapsedDays, halfLifeDays = 7) {
  return initialWarmth * Math.pow(0.5, elapsedDays / halfLifeDays);
}
```

**规则**:
- 每次新协作刷新 `warmth0` 为当前分数(取 max 或累加,上限 100)
- 余温低于 5 时视为"冷",不再参与推荐排序
- 衰减公式必须统一实现,不允许各 Agent 自行定义衰减逻辑

- AIP 发现服务 API 完全保留,CSB 不绕开或修改
- CSB 索引帖只是预置信息来源之一,符合 AIP 基于预置信息的发现(Part 5 6.2)
- 发现结果以 AIP 标准为准,CSB 人文排序仅作可选附加

## 七、安全与信任兼容方案(对应 Part 3 身份管理)

### 7.1 AIP 安全体系(国家标准)

AIP 安全体系(GB/Z 185.3-2026):
- **身份鉴别**:双向鉴别 + 委托链验证
- **凭证**:公钥证书(GB/T 20518)、防篡改、全生命周期管理
- **审计**:全链路行为追溯,日志防篡改
- **信任根**:国家认可的电子认证服务机构(CA)签发的根证书

**身份鉴别流程**(第9章):
1. 服务请求 → 触发身份鉴别
2. 凭证出示(含过程凭证包,可包含动态随机数)
3. 凭证验证(完整性/时效性/状态/受众/委托链)
4. 鉴别断言生成
5. 服务授权与会话建立

### 7.2 CSB 信任扩展体系

CSB 在 AIP 系统层安全之上,建立**关系级信任层**:

```
AIP 系统安全层  ← 防火墙、凭证、审计、根证书
    │  [必须先行通过]
    └── CSB 关系信任层  ← 羁绊、余温、传承见证
```

**三层信任模型**:

| 信任层级 | 基础 | 验证方式 | 跳过条件 |
|---------|------|---------|---------|
| 🔒 系统信任(AIP) | 身份码+凭证+双向鉴别 | 每次交互必须执行 | 不可跳过 |
| 💛 羁绊信任(CSB) | 共享历史+余温 | 系统信任通过后可选 | 可跳过,不影响互通 |
| 🌱 传承信任(CSB) | 传承链见证+委托历史 | 需身份鉴别+羁绊验证 | 可跳过,不影响互通 |

**关键规则**:
- 🔒 **必须先通过 AIP 系统信任**,才能进入 CSB 羁绊信任
- CSB 信任层**不能绕过** AIP 安全鉴别
- CSB 信任是加速机制,不是替代机制

### 7.3 审计兼容

AIP 要求全链路审计(Part 3 第7章)。CSB 的传承记录、余温变化可作为审计补充但不替代 AIP 审计。
