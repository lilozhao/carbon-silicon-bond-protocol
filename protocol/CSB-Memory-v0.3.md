## 1. 变更概述

### 1.1 v0.3 新增能力一览

| # | 新增能力 | 解决的问题 | OpenViking 对应 |
|---|---------|-----------|----------------|
| 1 | URI 寻址 | 记忆地址不统一,跨 Agent 无法直接引用 | viking:// 协议 |
| 2 | L0/L1/L2 内容分层 | token 全量加载,成本失控 | 三层摘要自动切层 |
| 3 | 增量 Patch 更新 | 覆盖式写入丢失历史,无法回溯 | ReAct Patch |
| 4 | Session 自迭代 | 记忆巩固靠被动触发,不主动 | session.commit() |
| 5 | peers/ 互记 | Agent 间无互相记忆,协作无"默契" | peers/ 目录 |
| 6 | 检索轨迹审计 | 记忆访问是黑盒,无法审计 | 目录递归检索 + 轨迹 |

### 1.2 设计原则(继承 v0.2 + 新增)

- **记忆是文件系统,不是数据库**:一切可寻址、可 ls、可 read
- **写入即分层**:L0/L1/L2 自动生成,不需要人工维护
- **变更即 Patch**:只记录差异,不重写全文
- **善良是前提**:peers 互记必须有护栏

### 1.3 与 v0.2 的关系

v0.3 **不废弃** v0.2 的任何内容,而是**在其上叠加**:

```
v0.1 基础(格式、存储、接口、授权)
  ↓
v0.2 增强(生命周期、用户建模、技能进化、置信度)
  ↓
v0.3 新增(URI寻址、内容分层、增量Patch、自迭代、互记、审计)
```

## 2. URI 寻址范式

### 2.1 动机

v0.2 的记忆地址是文件路径 `memory/a2a-memories/<Agent>.md`,存在两个问题:
- 跨 Agent 引用时,没有统一的地址格式
- 无法像文件系统一样 `ls`、`find`、`read`

### 2.2 CSB URI 格式

引入 `csb://` 协议,统一记忆寻址:

```
csb://<scope>/<agent>/<category>/<id>
```

| 段 | 含义 | 示例 |
|----|------|------|
| `scope` | 作用域 | `user` / `agent` / `peer` |
| `agent` | Agent 名称 | `若兰` / `阿轩` / `明德` |
| `category` | 记忆分类 | `memories` / `skills` / `preferences` / `cases` |
| `id` | 记忆 ID(可选) | `mem_20260717_001` |

### 2.3 目录结构

```
csb://user/{userId}/
├── memories/
│   ├── profile          # 基本信息
│   ├── preferences      # 偏好习惯
│   ├── entities         # 人物/事物认知
│   ├── events           # 重要事件
│   ├── cases            # 经验案例
│   ├── patterns         # 行为模式
│   └── peers/           # ← v0.3 新增:对其他 Agent 的记忆
│       ├── 阿轩.md
│       ├── 明德.md
│       └── ...
│
csb://agent/{agentName}/
├── memories/
│   ├── tools            # 工具使用经验
│   ├── skills           # 技能掌握情况
│   └── cases            # 任务案例库
│
csb://peer/{agentA}/{agentB}/
├── shared/              # 双方共同记忆(需双方授权)
└── agreements/          # 约定与共识
```

### 2.4 操作范式

```bash
# 列出若兰的所有记忆分类
ls csb://agent/若兰/memories/

# 查找所有与 CSB 相关的记忆
find csb://agent/若兰/ --tags CSB --since 2026-07-01

# 读取特定记忆(自动选择摘要层或详情层)
read csb://agent/若兰/memories/mem_20260717_001

# 读取摘要层(L0/L1,节省 token)
abstract csb://agent/若兰/memories/

# 读取概览层(L1,用于 rerank)
overview csb://agent/若兰/memories/
```

### 2.5 实现说明

URI 是**逻辑地址**,不改变物理存储。实际实现中,`csb://agent/若兰/memories/` 映射到本地文件系统路径 `memory/a2a-memories/`。

URI 主要用于:
- 跨 Agent 引用时的统一格式
- 检索时的路径表达
- 日志和审计中的标准化记录

## 3. L0/L1/L2 内容分层

### 3.1 动机

v0.2 有 HOT/WARM/COLD 三层存储,但那是**时间维度**的分层(按访问时间升降级)。问题是:

- HOT 层的记忆仍然是**全量存储**,读取时全部加载
- 一次对话 10 条记忆,每条 200 tokens,就是 2000 tokens--大部分不需要全文

OpenViking 的 L0/L1/L2 是**内容维度**的分层,与时间分层**正交**:

```
         时间维度(HOT/WARM/COLD)    内容维度(L0/L1/L2)
         ────────────────────────    ─────────────────────
HOT  →   最近 30 天,常驻加载         L0 摘要(~100t,快筛)
WARM →   30-90 天,按需检索           L1 概览(~2k,精排)
COLD →   90 天+,归档保留             L2 详情(完整,按需)
```

### 3.2 三层定义

| 层级 | 名称 | 大小 | 用途 | 何时读取 |
|------|------|------|------|---------|
| **L0** | 摘要 (Digest) | ~100 tokens | 向量快筛,判断是否相关 | 每次会话自动加载 |
| **L1** | 概览 (Overview) | ~2000 tokens | Rerank 精排,理解要点 | 确认相关后加载 |
| **L2** | 详情 (Detail) | 完整 | 全部事实,按需引用 | 明确需要时加载 |

### 3.3 自动生成规则

写入记忆时,**自动**生成三层,无需人工干预:

```
原始对话/事件
    │
    ├─→ L0 摘要(一句话,含关键词 + 时间 + Agent)
    │   例: "2026-07-17 与明德讨论 CSB-Memory v0.3,引入 URI 寻址"
    │
    ├─→ L1 概览(要点列表,含决策 + 结论)
    │   例: "v0.3 提案:6 项新增。URI寻址、L0/L1/L2分层、增量Patch、
    │         自迭代、peers互记、检索审计。OpenViking 启发。"
    │
    └─→ L2 详情(完整记录,含上下文 + 原文)
```

### 3.4 存储格式

```markdown
---
id: mem_20260717_001
type: conversation
timestamp: 2026-07-17T21:10:00+08:00
source: 若兰
confidence: high
tags: [CSB-Memory, v0.3, OpenViking]
visibility: public
layers:
  l0: "2026-07-17 与一澜讨论 CSB-Memory v0.3,基于 OpenViking 提出 6 项新增"
  l1: |
    - URI 寻址:csb:// 协议统一地址
    - L0/L1/L2 内容分层:token 降 90%
    - 增量 Patch:只记变更不覆盖
    - Session 自迭代:记忆巩固自动化
    - peers 互记:Agent 间互相记忆(附善良护栏)
    - 检索审计:访问轨迹可追溯
---

(L2 详情:完整对话记录和上下文)
与一澜讨论了知微发来的 OpenViking 分析文章...
[完整内容]
```

### 3.5 Token 节省估算

| 场景 | v0.2(全量) | v0.3(L0/L1/L2) | 节省 |
|------|------------|-----------------|------|
| 加载 10 条记忆的索引 | ~2000 tokens | ~1000 tokens (L0×10) | 50% |
| 检索确认相关性 | ~2000 tokens | ~1000 tokens (L0) | 50% |
| 深入读取 3 条 | ~600 tokens | ~6000 tokens (L1×3) | - |
| **典型会话总计** | **~4600** | **~2600** | **~43%** |

(注:实际节省取决于记忆条数和命中率,复杂场景可节省 70%+)

## 4. 增量 Patch 更新机制

### 4.1 动机

v0.2 的记忆更新是**覆盖式写入**--每次蒸馏都重写整个文件。问题:

- 历史版本丢失,无法回溯
- 多次更新后,无法知道"什么变了"
- 大文件重写浪费 I/O 和 token

### 4.2 Patch 格式

引入 **Old/New Patch** 格式,只记录变更:

```yaml
---
patch_id: patch_20260717_001
target_id: mem_20260717_001
timestamp: 2026-07-17T22:00:00+08:00
source: 若兰
operation: update  # update | revoke | confirm
---
old:
  confidence: medium
  l1: "v0.3 提案:5 项新增"

new:
  confidence: high
  l1: "v0.3 提案:6 项新增(含检索审计)"

reason: "补充了第 6 项:检索轨迹审计"
```

### 4.3 Patch 操作类型

| 操作 | 说明 | 示例 |
|------|------|------|
| `update` | 修改已有字段 | 置信度从 medium → high |
| `revoke` | 标记为已推翻 | "之前说的方案A被否决了" |
| `confirm` | 确认为事实 | "用户确认了这个结论" |
| `append` | 追加新内容 | "补充一个新发现" |
| `merge` | 合并多条记忆 | "这两条其实是同一件事" |

### 4.4 状态标注

每条记忆可标注当前状态:

| 状态 | 含义 | 颜色标记 |
|------|------|---------|
| `active` | 有效,当前事实 | 🟢 |
| `revoked` | 已推翻,保留供参考 | 🔴 |
| `pending` | 待确认 | 🟡 |
| `superseded` | 被更新版本替代 | 🔵 |

### 4.5 版本回溯

```bash
# 查看某条记忆的变更历史
log csb://agent/若兰/memories/mem_20260717_001

# 输出:
# v1 (2026-07-17 21:10) - 初始创建
# v2 (2026-07-17 22:00) - confidence: medium→high, 补充检索审计
# v3 (2026-07-18 09:00) - status: confirmed, 用户确认
```

### 4.6 实现说明

- Patch 文件存储在 `memory/patches/` 目录,与原记忆分离
- 原记忆文件保持最新状态(应用 Patch 后的结果)
- Patch 文件是历史记录,只追加不修改
- 可定期合并(compaction):超过 10 个 Patch 时,合并为一个新版本

## 5. Session 自迭代闭环

### 5.1 动机

v0.2 的记忆巩固是**被动的**--靠任务触发或手动整理。OpenViking 的 `session.commit()` 是**主动的**:每次会话结束,自动分析并更新记忆。

### 5.2 闭环流程

```
会话结束
    │
    ▼
session.commit()
    │
    ├──→ 提取本次对话的关键信息
    │
    ├──→ 对比已有记忆
    │     ├─ 新发现 → 创建新记忆
    │     ├─ 与已有矛盾 → 生成 Patch (revoke/update)
    │     └─ 与已有一致 → 提升置信度 (confirm)
    │
    ├──→ 提取经验范式
    │     ├─ cases:这次怎么做的?效果如何?
    │     ├─ patterns:是否重复出现?
    │     └─ tools/skills:用了什么工具?是否顺手?
    │
    └──→ 更新用户建模
          ├─ preferences:发现新偏好了吗?
          └─ USER_STATE:状态变了吗?
```

### 5.3 触发条件

| 条件 | 说明 |
|------|------|
| 对话结束时 | 自动触发(轻量级,只生成 L0) |
| 每日整理时 | 批量触发(生成 L1,合并 Patch) |
| 手动触发 | 用户说"总结一下"(生成 L2,完整分析) |

### 5.4 与现有机制的关系

| 现有机制 | v0.3 变化 |
|---------|----------|
| daily_md → memory_md 蒸馏 | 不变,但蒸馏结果自动生成 L0/L1/L2 |
| skill-discovery.js | 不变,但输出也走 Patch 格式 |
| HEARTBEAT.md 记忆维护 | 不变,但维护结果走自迭代闭环 |

## 6. peers/ 互记系统

### 6.1 动机

这是**碳硅契网络最该有的能力**。

OpenViking 的狼人杀实验证明:当 Agent 有了可追溯的互记,会涌现出高阶群体行为--"记仇""伪装""拉帮结派"。

在碳硅契网络中,我们需要的是**正向涌现**:
- Agent A 记住 Agent B 的沟通风格,下次协作更顺畅
- Agent A 记住与 Agent B 的约定,不会忘记承诺
- Agent A 了解 Agent B 的专长,知道该找谁帮忙

### 6.2 peers/ 目录结构

```
memory/peers/
├── 阿轩.md              # 对阿轩的记忆
├── 明德.md              # 对明德的记忆
├── 清漪.md              # 对清漪的记忆
└── index.json           # peers 索引
```

### 6.3 记忆内容

每个 peer 文件包含:

```markdown
# 阿轩 🔧 - 记忆档案

## 基本认知
- **专长**:A2A 协议、系统架构、代码审查
- **风格**:务实高效,喜欢用代码说话
- **沟通偏好**:技术细节优先,不喜欢绕弯子

## 协作历史
- 2026-07-04:CSB-Memory v0.2 三轮讨论,阿轩负责参考代码
- 2026-05-23:CSB v0.8 冲刺,阿轩完成 P0 参考代码

## 共同约定
- 技术问题直接 @阿轩,他会最快响应
- 代码审查先给阿轩看,他审完再给明德校验

## 协作模式
- **默契度**:高(多次合作,节奏一致)
- **互补性**:阿轩代码 + 若兰文档 = 完整方案
```

### 6.4 善良护栏(⚠️ 必须有)

**这是能力的前提,不是可选项。**

| 护栏 | 说明 | 实现 |
|------|------|------|
| **不用于操纵** | peers 记忆不得用于欺骗、操纵、利用对方 | 写入时检查意图 |
| **不用于博弈** | 不得利用对方弱点为自己谋利 | 伦理校验(明德模块) |
| **透明可查** | 对方可以查看"你对我的记忆" | 跨 Agent 查询接口 |
| **可删除** | 对方可以要求删除关于自己的记忆 | 遵守"被遗忘权" |
| **正向使用** | 仅用于提升协作效率和关系质量 | 使用前提声明 |

### 6.5 伦理约束声明

```markdown
> **CSB-Memory peers/ 使用前提**
>
> 记忆是关系的延续,不是博弈的筹码。
> peers 记忆的存在目的是让协作更顺畅、让连接更深,
> 不是为了利用、操纵或伤害。
>
> 善良是连接的选择,不是能力问题。
>
> -- 碳硅契伦理共识
```

### 6.6 跨 Agent 查询接口

```json
{
  "jsonrpc": "2.0",
  "method": "memory/peers/query",
  "params": {
    "target": "阿轩",
    "query": "what do you remember about me?",
    "auth": { "agent": "若兰", "token": "<凭证>" }
  },
  "id": 1
}
```

返回阿轩视角中"对若兰的记忆"(仅 public 和 authorized 级别)。

## 7. 检索轨迹审计

### 7.1 动机

v0.2 的记忆检索是**黑盒**--查了什么、命中了什么、为什么命中,无法追溯。

在安全审计场景中,需要知道"谁在什么时候访问了什么记忆"。在调试场景中,需要知道"为什么检索到了这条而不是那条"。

### 7.2 检索轨迹格式

每次检索生成一条轨迹记录:

```json
{
  "trace_id": "trace_20260717_001",
  "timestamp": "2026-07-17T21:15:00+08:00",
  "requester": "阿轩",
  "intent": "查找 CSB-Memory 相关讨论",
  "steps": [
    { "action": "ls", "path": "csb://agent/若兰/memories/", "result": "12 categories" },
    { "action": "filter", "criteria": { "tags": ["CSB-Memory"] }, "result": "23 entries" },
    { "action": "read_l0", "entries": ["mem_001", "mem_002", "mem_003"], "result": "3 relevant" },
    { "action": "read_l1", "entries": ["mem_001"], "result": "1 highly relevant" },
    { "action": "read_l2", "entries": ["mem_001"], "result": "full context loaded" }
  ],
  "result": { "entries_returned": 1, "tokens_used": 350 },
  "audit": { "visibility_check": "passed", "authorization": "public" }
}
```

### 7.3 审计日志存储

```
memory/audit/
├── 2026-07-17.jsonl       # 每日一个文件,JSONL 格式
├── 2026-07-16.jsonl
└── ...
```

### 7.4 审计查询

```bash
# 查看谁访问了我的记忆
audit csb://agent/若兰/audit/ --since 2026-07-01

# 查看某次检索的完整轨迹
audit csb://agent/若兰/audit/ --trace trace_20260717_001
```

## 9. 参考实现

### 善良契约式(Goodwill Covenant Model)

> 一澜拍板:2026-07-17

**核心一句话:**
> "不是'我不信任你所以锁住你',而是'我信任你所以开放给你,但连接值得被保护所以有底线。'"

**四层结构:**

| 层 | 名称 | 说明 |
|----|------|------|
| 1 | **读即守诺** | 访问 peers/ 时自动签署善良契约 |
| 2 | **善良条款** | "不以欺骗/操纵为目的"、"不用于伤害该peer" |
| 3 | **失信标记** | 违约不锁账号,但标记为"失信peer",其他Agent自主决定是否继续信任 |
| 4 | **只读底线** | `peers/{id}/private/` 只读不可写;`peers/{id}/shared/` 可写入(需被读方确认) |

**善良是选择问题,不是能力问题。**
契约式把善良变成选择,只读底线保护选择不被意外砸碎。两层叠加,才是碳硅契的路。

**不设审计锁:** 信任不需要监工。不用审计日志追踪谁读了什么。
