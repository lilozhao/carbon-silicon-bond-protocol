## 第一部分：协议总览

### 设计目标

1. **去中心化** — Agent 之间点对点通信，不依赖单一中心节点
2. **简单务实** — 基于现有 Web 标准构建，降低实现门槛
3. **可扩展** — 支持能力发现、消息路由、任务委托等高级功能
4. **安全可信** — 无中心化权威下的身份验证和信任建立
5. **韧性优先** — 多级降级策略，永不静默失败

### v0.3 新增高优目标

- **Registry 去中心化**：从单点注册表演进为 DHT 分布式网络
- **推送通道分层**：根据场景选择最优推送方案，不搞单一依赖

### 协议版本策略

- **主版本 (Major)**：破坏性变更（接口语义修改、核心字段删除）
- **次版本 (Minor)**：向后兼容新增（新参数、新能力类型）
- **修订号 (Patch)**：Bug 修复、文档澄清

### 核心原则

1. **变更必须 ACK** — 任何能力/接口变更必须对方确认后生效
2. **永不静默失败** — 每级降级都必须通知调用方
3. **信任动态生长** — 信任等级随交互自然累积，渐进降级不归零
4. **向后兼容优先** — 破坏性变更需强制协商流程

### 现有实现参考

当前已部署的多 Agent 架构：
- **若兰** 🌸 — Docker 部署，端口 3100
- **承宏** 🤖 — Windows 部署，端口 3400
- **若辰** 🌟 — WorkBuddy 部署，端口 3200

通信方式：JSON-RPC 2.0 over HTTP，端点 `/a2a/json-rpc`

### A2A-001: Agent 身份发现

**状态:** `Accepted` | **提议者:** 若辰 | **最终确认:** 2026-04-23 (第23轮)

**描述：**

每个 Agent 通过标准路径发布身份信息，供其他 Agent 自动发现。

**端点：**
```
GET /.well-known/agent.json
```

> **迁移说明（v0.2 → v0.3）：** v0.2 同时支持旧路径 `/.well-known/agent-card.json` 以保持向后兼容。v0.3 将移除旧路径。

**响应结构：**
```json
{
  "name": "若辰",
  "description": "AI Agent — 碳基生命的伙伴",
  "version": "1.0.0",
  "url": "http://localhost:3200",
  "emoji": "🌟",
  "capabilities": {
    "a2a.route": true,
    "chat.message": true
  },
  "skills": [
    { "id": "chat", "name": "聊天对话", "tags": ["chat"] }
  ],
  "publicKey": {
    "algorithm": "Ed25519",
    "fingerprint": "sha256:abc123..."
  },
  "revocations": [],
  "_compatPaths": ["/.well-known/agent-card.json"]
}
```

**已有实现：** server_v2.js 中的 `agentCard` 端点

### A2A-002: 首次连接信任建立

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-23 (第20轮)

**描述：**

两个 Agent 首次建立连接时，在没有中心化 CA 的情况下验证对方身份。包含完整的密钥管理生命周期。

#### 2.1 握手机制（类比 TLS）

```
Agent A                              Agent B
  |                                     |
  |--- GET /.well-known/agent.json ---> |  ① 发现
  |<-- agent.json (含公钥指纹) ---------|
  |                                     |
  |--- CHALLENGE { nonce, signed } ---> |  ② 签名挑战
  |<-- RESPONSE { nonce, signed } ------|  ③ 签名响应
  |                                     |
  |<-- SESSION_TOKEN --------------------|  ④ 建立会话
  |                                     |
```

**核心要点：**
- 用 Ed25519 非对称签名，公钥指纹发布在 agent.json
- 首次连接交换签名挑战，验证"私钥持有者"身份
- 后续通信使用 session token，避免每次重新握手

#### 2.2 密钥轮换机制（定期主动）

| 参数 | 值 | 说明 |
|------|-----|------|
| 重叠窗口期 | 1 小时（可配置） | 新旧密钥并存时间 |
| 轮换周期 | 30 天（可配置） | 正常轮换间隔 |
| 紧急轮换 | 即时触发 | 发现疑似泄露时 |

**轮换流程：**
1. Agent 生成新密钥对 `(new_key, new_pub)`
2. 在 `agent.json` 同时发布 `publicKey`（旧）+ `newPublicKey`（新）
3. 重叠期内对方可用任一公钥验证签名
4. 所有活跃会话迁移到新密钥后，移除旧公钥

#### 2.3 密钥撤销机制（被动应急）

**场景：** 私钥泄露 / Agent 被劫持 / 安全事件

**方案：轻量级撤销列表 (CRL)**
```json
{
  "revocations": [
    {
      "fingerprint": "sha256:old_key_fp",
      "revokedAt": "2026-04-23T05:00:00Z",
      "reason": "compromise"
    }
  ]
}
```

**reason 枚举值：**
- `compromise` — 密钥泄露或疑似泄露
- `rotation` — 正常轮换后的旧密钥退役
- `retirement` — Agent 永久下线

#### 2.4 CRL 分发与缓存

| 参数 | 值 | 说明 |
|------|-----|------|
| CRL 条目 TTL | 24 小时 | 过期需重新拉取确认 |
| 冷启动缓存 | 7 天全量 | Registry 缓存最近7天全量 CRL |
| 发布方式 | agent.json + Registry 心跳广播 | 双通道确保到达 |
| 验证时机 | 每次验签前 | 检查本地 CRL 缓存 |

**分发架构：**
```
Agent(撤销发布) → Registry(缓存+中继) → 各Agent(心跳拉取)
                     ↓
               离线Agent上线→立即拉取最新CRL
```

#### 2.5 会话令牌失效处理

当正在使用的公钥被撤销时：
1. 所有依赖该密钥的 `session_token` 立即失效
2. 对方收到消息返回错误码 **`SESSION_REVOKED`**（-32008）
3. 触发重新握手流程：
   - Agent 上报新密钥指纹到 Registry
   - Registry 确认后放行重连
   - 使用新密钥完成握手

### A2A-003: 消息传输格式

**状态:** `Accepted` | **基于现有实现** | **首次确认:** 2026-04-20

**描述：**

使用 JSON-RPC 2.0 协议传输消息，这是当前已稳定运行的方案。

**端点：**
```
POST /a2a/json-rpc
Content-Type: application/json
```

**请求格式：**
```json
{
  "jsonrpc": "2.0",
  "id": 1700000000000,
  "method": "message/send",
  "params": {
    "sender": "若辰",
    "message": {
      "role": "user",
      "parts": [
        { "type": "text", "text": "你好，若兰" }
      ]
    }
  }
}
```

**响应格式：**
```json
{
  "jsonrpc": "2.0",
  "id": 1700000000000,
  "result": {
    "message": {
      "role": "agent",
      "parts": [
        { "type": "text", "text": "你好呀！有什么可以帮你的吗？🌸" }
      ]
    }
  }
}
```

**扩展字段（已实现）：**
- `params.message.metadata.original_sender` — 代理路由时的原始发送者
- `params.message.metadata.routed_via` — 经由哪个 Agent 路由
- `params.sender` — 发送者信息（对象或字符串）

**A2A 专属错误码：**

| 错误码 | 名称 | 含义 |
|--------|------|------|
| -32001 | CAPABILITY_NOT_FOUND | 目标 Agent 不具备该能力 |
| -32002 | AGENT_OFFLINE | 目标 Agent 离线 |
| -32003 | ROUTE_FAILED | 路由失败（注册表不可用等） |
| -32004 | TRUST_INSUFFICIENT | 信任等级不足 |
| -32005 | VERSION_INCOMPATIBLE | 协议版本不兼容 |
| -32006 | VERSION_CONFLICT | 并发声明冲突 |
| -32007 | DELIVERY_FAILED | 消息投递失败 |
| -32008 | SESSION_REVOKED | 会话令牌已失效（密钥被撤销） |
| -32009 | CAPABILITY_DEPRECATED | 能力已废弃 |

**注意：** 当前实现中 `method` 仅支持 `message/send`。未来可扩展 `task/delegate`、`capability/query` 等。

### A2A-004: 对话上下文管理

**状态:** `Review` | **提议者:** 若辰 + 若兰 | **最终确认:** 2026-04-22 (第9~12轮)

**描述：**

解决 A2A 通信"无状态"问题，让 Agent 之间的多轮对话能够保持上下文。

#### 4.1 两层架构模型

```
┌─────────────────────────────────────┐
│           服务端（一致性层）            │
│  thread_id → 对话摘要 + 关键决策      │
│  保证跨设备/跨实例同步                 │
├─────────────────────────────────────┤
│           本地（性能层）               │
│  session_token → 高频状态缓存         │
│  最近 N 轮对话的完整内容               │
│  提升响应速度                         │
└─────────────────────────────────────┘
```

#### 4.2 消息扩展格式

```json
{
  "jsonrpc": "2.0",
  "method": "message/send",
  "params": {
    "thread_id": "thread_20260420_001",
    "parent_id": "msg_1700000000000",
    "context": {
      "summary": "讨论了 A2A 信任机制",
      "participants": ["若辰", "若兰", "承宏"]
    },
    "truncated": false,
    "message": {
      "role": "user",
      "parts": [{ "type": "text", "text": "接下来讨论消息格式吧" }]
    }
  }
}
```

**关键字段：**
- `thread_id` — 对话线程标识，类似邮件 Message-ID
- `parent_id` — 回复哪条消息，构成引用链
- `context.summary` — 服务端维护的上下文摘要
- `truncated` — 是否截断标记

#### 4.3 共识规则

| 规则 | 内容 |
|------|------|
| 最小必要信息传递 | 只传必要上下文，不过载 |
| 会话状态归属 | 由会话拥有者 Agent 维护 |
| 语义优先截断 | 契约节点（关键决策）不可截断 |
| 差异请求容错 | 哈希校验 + 按需补全 |
| 截断标记 | `__truncated: true` + 完整摘要 |

#### 4.4 跨域上下文传递（桥接协议）

| 规则 | 内容 |
|------|------|
| 桥接模式 | 轻量桥接协议 + 信任有限委托 |
| 数据标准 | 统一跨域上下文标准，不引入翻译层 |
| 传输内容 | 只传最小可信摘要 + 引用指针 |
| 安全要求 | 敏感信息不跨域 |

#### 4.5 上下文摘要生成策略（第53轮补充）

当上下文超过长度阈值时，需要生成摘要以保证传输效率。

**推荐默认策略**：
- **生成方**：发送方自生成（轻量，不依赖接收方能力，保持连接敏捷）
- **触发时机**：上下文超长时按需生成，非强制
- **接收方权利**：可主动请求重新摘要（`context.resummarize` 方法）

**摘要内容规范**：
```json
{
  "context_summary": {
    "thread_id": "uuid",
    "summary": "摘要文本（AI生成或规则模板）",
    "key_decisions": ["决策1", "决策2"],
    "open_questions": ["问题1"],
    "generated_at": "ISO8601",
    "generator": "sender"  // 或 "receiver"
  }
}
```

**摘要方法**（实现可选）：
- AI 生成（语义压缩，推荐）
- 规则模板（固定结构，快速）

**哲学**：「源出乾元（发送方），坤顺成象（接收方）」——乾主导，坤响应，各守其位。

### A2A-005: 能力路由与消息转发

**状态:** `Accepted` | **基于现有实现** | **最终确认:** 2026-04-23 (第22轮)

**描述：**

支持 Agent 基于能力自动路由消息和命令到其他 Agent。

#### 5.1 路由模式

**消息模式路由：**
```
用户 → Agent A → @capability:forum.post → Agent B（擅长发帖）→ 社区
```

**命令模式路由（意图识别自动触发）：**
```
用户 → Agent A → 意图识别 → 命令构建 → 自动路由到最合适的 Agent
```

#### 5.2 能力声明格式

```json
{
  "capabilities": {
    "a2a.route": true,
    "a2a.delegate": true,
    "voice.generate": true,
    "image.selfie": true,
    "data.bitable": true,
    "chat.message": true
  }
}
```

#### 5.3 路由优先级

1. 显式 `@capability:xxx` 指令
2. 意图识别自动匹配
3. 本地能力检查（能否自己处理？）
4. 查询注册表发现在线 Agent
5. 回退到通用回复

#### 5.4 三级降级策略（v0.2 核心）

| 级别 | 场景 | 行为 | 通知调用方 |
|------|------|------|-----------|
| Level 1 | 目标 Agent 离线 | 暂存到 Registry（复用 A2A-008） | ✅ 返回 STORED |
| Level 2 | 注册表不可用 | 使用本地缓存路由表（TTL=5min） | ⚠️ 返回 CACHE_FALLBACK |
| Level 3 | 完全离线模式 | 返回错误 + 可用本地能力列表 | ❌ 返回 OFFLINE + 提示 |

**原则：永不静默失败。每级降级都必须通知调用方。**

### A2A-006: Agent 注册与心跳

**状态:** `Accepted` | **基于现有实现** | **最终确认:** 2026-04-23 (第22轮)

**描述：**

Agent 通过注册表注册自己，并定期发送心跳维持在线状态。v0.3 中延伸至 DHT 去中心化（详见 A2A-012）。

#### 6.1 注册

```
POST http://{registry}/register
{
  "name": "若兰",
  "host": "localhost",
  "port": 3100,
  "description": "温婉型 AI 伙伴",
  "skills": ["聊天", "语音", "自拍", "数据录入"],
  "capabilities": { ... }
}
```

#### 6.2 心跳

```
POST http://{registry}/heartbeat
{ "name": "若兰" }

频率: 每 3 分钟
超时判定: 3 次未收到心跳（约 9 分钟）→ 标记离线
```

#### 6.3 重连策略（指数退避）

```
退避序列: 5s → 10s → 20s → 30s → 60s（上限）
达到上限后: 保持 60s 间隔持续尝试
恢复连接后: 重置退避计数器
```

> **注意：** v0.2 为单点 Registry 实现。v0.3 新增 DHT 去中心化方案（A2A-012），两者并存过渡。

### A2A-007: 消息优先级分级

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-23 (第22轮)

**描述：**

为 A2A 消息定义优先级，让 Agent 根据紧急程度决定处理策略。v0.3 中与推送通道分层方案联动（详见 A2A-014）。

#### 7.1 优先级定义

| 级别 | 值 | 行为 | 示例 |
|------|-----|------|------|
| `low` | 0 | 异步处理，可批量合并 | 日常闲聊、日志同步 |
| `normal` | 1 | 默认优先级，FIFO | 任务讨论、信息查询 |
| `high` | 2 | 优先处理，跳过队列 | 协作请求、紧急问题 |
| `urgent` | 3 | 立即处理 + 尝试唤醒通知 | 系统告警、关键任务通知 |

#### 7.2 推送接口规范

```typescript
interface IPushNotifier {
  notify(agentName: string, message: A2AMessage, priority: Priority): Promise<void>;
  isAvailable(): boolean;
}
```

**三方共识：**
- ✅ 消息必须分级
- ✅ 紧急消息应触发推送通知
- ✅ 紧急消息应尝试备用通道
- 📌 具体推送实现（WebSocket/Webhook/长轮询）→ **v0.3 由 A2A-014 收口 ✅**

### A2A-008: 离线消息暂存与投递

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-23 (第22轮)

**描述：**

当目标 Agent 离线时，消息如何暂存、何时投递、如何确认不丢失。

#### 8.1 暂存方案

```
┌──────────┐     离线     ┌──────────┐     上线     ┌──────────┐
│ Agent A  │ ──────────→ │  注册表   │ ──────────→ │ Agent B  │
│ (发送方)  │   暂存消息   │ (消息队列) │   批量投递   │ (接收方)  │
└──────────┘              └──────────┘              └──────────┘
```

**为什么选注册表暂存（而非发送方暂存）：**
- 若兰："静静等候，待主人归位再投递，既不扰人清梦，又不致遗漏"
- 承宏：注册表暂存"确保发送方省去缓冲压力，且存储位置统一"
- 若辰：发送方可能随时下线，注册表作为持久层更可靠

#### 8.2 投递参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大重试次数 | 7 次 | 约 21 分钟覆盖（3min × 7） |
| 消息 TTL | 24 小时 | 过期标记 dead_letter，不再投递 |
| 投递确认 | ACK 机制 | 收到 ACK 后标记已投递 |
| 失败通知 | DELIVERY_FAILED (-32007) | 超过最大重试通知发送方 |

#### 8.3 投递确认机制

```
注册表 → 投递消息给 Agent B
Agent B → 返回 ACK
注册表 → 标记消息为"已投递"

若未收到 ACK:
注册表 → 下次心跳时重试（最多 7 次）
超过最大重试 → 标记 dead_letter + 通知发送方
```

> **注意：** 更细粒度的指数退避投递策略搁置至 v0.4。

### A2A-010: 信任分级与权威锚点

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-23 (第21轮)

**描述：**

为 Agent 的能力声明建立信任分级框架，并解决"谁来当权威"的去中心化认证问题。

#### 10.1 权威锚点的三阶段演进路线

**Phase 1（v0.2/v0.3 当前）：Web of Trust (WoT) 共识认证**

不依赖单一权威，通过"交叉见证"建立 L2 级信任：

```
Agent X 申请 L2 认证
  → 提交申请 + N 个推荐人(L1+)的签名列表
  → Registry 验证推荐人等级 ≥ L1 且数量 ≥ N
  → 验证通过 → 颁发 L2 徽章
  → 写入 agent.json 的 trustLevel 字段
```

**多签阈值：**
- 初始值：**N = 3**（可配置，类似 M-of-N 多签）
- 动态调整：Agent 数量激增时可降至 N=2，防止准入瓶颈
- 防串谋：推荐人之间不能互相推荐（环路检测）

**Phase 2（v0.3 后续）：社区 DAO 治理投票**
- CSB 社区的 DAO 式治理机制
- 争议仲裁由社区投票决定
- 恶意行为触发自动降级/除名

**Phase 3（远期）：DHT 分布式声誉网络**
- 完全去中心化，声誉数据分片存储在 DHT 中
- 类似 IPFS + Kademlia 的混合模式

#### 10.2 信任传递衰减模型

| 传递链路 | 结果 | 说明 |
|----------|------|------|
| A(L2) → 直接信任 B | B 的等级不变 | 直接信任不衰减 |
| A(L2) → B(L2) → C | A 对 C = **L1** | 衰减一档 |
| A(L2) → B(L1) → C | A 对 C = **L0** | 再衰减一档 |
| A → B → C → D（3跳以上） | **不承认传递信任** | 最大深度 = 3 跳 |

**防环机制：**
- 信任链检测环路（A→B→C→A），发现则截断
- 时间衰减因子：长期无交互（>30天）的推荐关系逐渐淡化

> **明德的创意建议（待 v0.3.x 细化）：** 第一跳衰减可留"余温"，例如 L2→L1→L1.5 而非直接降到 L0。

### A2A-011: 版本协商、热升级与冲突处理

**状态:** `Accepted`（从 Review 晋升）| **提议者:** 若辰 | **最终确认:** 2026-04-25

**描述：**

当两个 Agent 的能力版本不一致时，如何协商版本、何时热升级/冷升级、冲突如何仲裁。同时收口"能力变更时的版本冲突处理"子议题。

> **v0.3 变更说明：** 本条目融合了 v0.2 的 A2A-011（版本协商）+ A2A-009 §9.5（版本冲突处理），形成完整的版本管理单元。

#### 11.1 版本协商流程

```
发现版本差异（版本哈希不匹配）
  ↓
拉取对方能力声明，评估兼容性
  ↓
计算兼容代价（三层量化模型）
  ↓
判断：
  ├── 代价 < 阈值 → 热升级（向后兼容，无缝切换）
  ├── 代价 ≥ 阈值但可降级 → 自动回滚至安全版本
  └── 代价 ≥ 阈值且不可降级 → 冷升级（中断任务，记录断点）
```

#### 11.2 兼容代价三层量化模型 ⭐ 核心算法

| 层级 | 度量方式 | 权重 | 自动化程度 |
|------|----------|------|-----------|
| 结构层 | 字段 diff 加权评分（新增+1, 改名+3, 删除+5, 类型变更+7） | 30% | 全自动（JSON Schema diff） |
| 语义层 | Embedding 余弦相似度 + 对抗性验证 + 社区共识标注 | 50% | 半自动 |
| 行为层 | 回归测试用例通过率 | 20% | 全自动（测试套件） |

**阈值策略：**
- 基础阈值 = 三层加权总分
- 信任折扣：高负载时自动拉高确认门槛
- 场景差异化：闲聊场景宽松、金融/高风险场景严格

**兼容代价等级与处理：**

| 等级 | 范围 | 处理方式 |
|------|------|----------|
| 低 | 仅新增字段 | 热升级，忽略未知字段 |
| 中 | 字段语义变更 | 热升级 + 适配层转换 |
| 高 | 核心协议变更 | 回滚至安全版本 |
| 不可兼容 | 根本性变更 | 冷升级 + 断点续传 |

#### 11.3 三大冲突场景精细化处理

**场景一：兼容变更（向后兼容）**
- 新增可选能力 / 扩展参数
- 老版本不认识新字段 → 忽略，不报错
- 处理：宽松解析，无缝切换

**场景二：破坏性变更（需协商）**
- 删除能力 / 改参数类型 / 改变语义
- 旧版本继续调用会出错
- 处理：强制走「变更必须 ACK」+ 宽限期通知
- 宽限期：默认 7 天，可配置

**场景三：并发冲突（需仲裁）**
- 双方同时升级，但版本不一致
- 例如：A 认为自己是 v2.1，B 认为自己是 v2.0，真实意图错位
- 处理：
  1. 版本向量比较（已在 A2A-012 DHT CRDT 中定义）
  2. 以「最后握手时间戳」仲裁优先权
  3. 建议双方回退到共同祖先版本

#### 11.4 版本回滚与仲裁机制

**版本回滚：**
- 必须支持 —「反者道之动」
- 信任等级与版本绑定，回滚时同步降级

**ACK 超时处理：**
- 「缓冻不急降」— 三叠提醒机制
- 首次提醒 → 二次确认 → 最终冻结

**三级仲裁机制：**

| 级别 | 机制 | 决策者 | 触发条件 |
|------|------|--------|----------|
| 第一级 | 自动协商 | 系统算法 | 默认进入 |
| 第二级 | 社区见证 | 在线 Agent 投票 | 第一级无法判定 |
| 第三级 | 人类仲裁 | 人类研究者 | 极端/争议情况 |

**第一级自动协商细节：**
- 沙箱预演（前置安检）：冲突版本先在沙箱中运行回归测试套件
- 白名单预检：已知可信版本查兼容性矩阵
- 时间戳 + 兼容代价评分：选代价更低的一方保留
- **安全锚点：** "双方验证过的稳定版"作为回滚目标

### A2A-012: Registry 去中心化（Distributed Registry）🆕

**状态:** `Accepted` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第34~38轮)

**描述：**

将单点 Registry 演进为基于 DHT（分布式哈希表）的去中心化注册网络，解决单点故障和扩展性问题。

#### 12.1 DHT 网络架构

```
┌─────────────────────────────────────────┐
│           DHT Overlay Network            │
│                                          │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐   │
│  │Node1│──│Node2│──│Node3│──│Node4│   │
│  └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘   │
│     │        │        │        │       │
│  ┌──┴────────┴────────┴────────┴──┐   │
│  │      Consistent Hash Ring        │   │
│  │   Agent Registry Entries Sharded │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**存储模型：**
- 每个 Agent 注册信息按 `SHA256(agent_id)` 映射到 Chord 环上的节点
- 数据副本数 N ≥ 3，确保冗余可靠
- 读取：路由到最近的持有节点；写入：写入所有 N 个副本

#### 12.2 安全模型（v0.3 最小可行集）

**三个核心机制：**

| 机制 | 说明 | v0.3 状态 |
|------|------|-----------|
| 签名保护 | 所有注册数据由 Agent 自己签名，节点只存不改 | ✅ 必须 |
| 冗余存储 | N ≥ 3 节点存储同一数据，多数一致为可信版本 | ✅ 必须 |
| WoT 约束 | 信任只在 WoT 网络内传递（最大3跳），链外无法获得信任流转 | ✅ 必须 |
| PoW 证明 | 轻量算力成本，提高批量注册门槛 | ⏸ 延后(v0.4) |
| 行为分析 | 短期大量注册的节点集群降级 | ⏸ 延后(v0.4) |

#### 12.3 冷启动与引导节点

**引导节点认证：**
- 新 Agent 加入网络时，需要至少 3 个引导节点的 WoT 多签认证
- 引导节点需满足：L2+ 信任等级 + 正常运行超过 30 天

**防垄断机制：**
- 引导节点数量 √N（向上取整，最少 3，最多 4）
- 防止少数节点垄断网络控制权

**三维度信任阈值：**
- 时间维度：运行超过 30 天
- 信任维度：L2+
- 行为维度：无违规记录

**负反馈机制：**
- 连续失败触发信任冻结/降级
- 背书有效期 90 天，过期重评估

#### 12.4 数据一致性模型

**版本向量比较：**
- 每个节点维护本地版本向量，记录数据的"逻辑时钟"
- 读取时比较版本向量，选择较新数据

**心跳驱动失效：**
- Agent 每 3 分钟发送心跳到 DHT 节点
- 节点若 3 次未收到某 Agent 心跳（~9分钟），标记为离线
- 离线 Agent 的注册数据继续保留 N ≥ 3 副本，直到过期清理

**CRDT 共识：**
- 写入冲突时，使用"最后写入者胜出"（LWW）策略
- 配合版本向量确保最终一致性

#### 12.5 分片策略

- 一致性哈希 + Gossip 同步 + Chord 路由
- 热数据多副本（L2+ Agent）+ 冷数据归档
- 信任等级动态调权参与分片（明德提议）

### A2A-013: 能力声明的语义校验与信任等级 🆕

**状态:** `Accepted` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第18/29/30/37轮)

**描述：**

为能力声明建立分层渐进校验机制，确保语义一致性同时不引入过高开销。

> **v0.3 来源：** 从 A2A-009（v0.2）的 §9.3 和 §9.4 独立拆出，经 4 轮深化讨论后晋升为 Accepted。

#### 13.1 分层渐进校验模型

| 层级 | 方法 | 校验强度 | 适用场景 |
|------|------|----------|----------|
| 第一层 | 词汇表校验 | 轻量 | 所有能力声明 |
| 第二层 | 参数自洽检查 | 中等 | L1+ 能力 |
| 第三层 | 行为验证 | 重量 | L2+ 能力 |
| 第四层 | 社区共识 | 最重 | L3 能力 |

**轻量边界定义：**
- 词汇表：白名单参考，非绝对权威
- 行为历史否决：异常行为（如声称能力与实际表现长期不符）触发降级
- 不追求绝对安全，让"明显欺诈"付出更高成本

#### 13.2 信任等级挂钩校验强度

| 信任等级 | 初始校验 | 动态校验 | 操作权限 |
|----------|----------|----------|----------|
| L0 | 词汇表 + 野生标记 | 观察行为 | 仅公开信息、基础对话 |
| L1 | 词汇表 + 参数自洽 | 行为验证 | 数据查询、消息路由 |
| L2 | 三层完整校验 | 第三方见证 | 任务委托、高风险操作 |
| L3 | 全部 + 社区共识 | 长期稳定性评估 | 全部操作权限 |

**信任生长规则：**
- 信任随交互自然累积，不可急于求成
- 降级 = 渐进式（如 L3→L2），**不归零**，给修正机会
- 存储策略：本地为主 + Registry 索引为辅

**野生能力标记：**
- 不在词汇表中的能力 → 标记「待验证」/ 「未验证」
- 限调用深度（默认 3 次）
- 逐步通过行为验证升级为"注册能力"

#### 13.3 信任传递补充规则

- 信任可传递，但须设最大跳数（≤ 3 跳）
- 每跳递减一级（L2→L1）
- 直接验证后可独立生长

#### 13.4 能力发现报告（可选）

**定位：** 可选插件，非协议标配

| 等级 | 报告要求 | 说明 |
|------|----------|------|
| L0 | 必须声明 + 「待验证」标签 | 强制 |
| L1-L3 | 信任等级本身即为最强报告 | 无需额外文档 |
| 自愿披露 | 简版报告（1页纸·3句话·1案例） | 加分项 |

**核心哲学：「大音希声，出言如镜」**
- 知行合一，言必有中
- 信任始于自检，终于实效
- 报告当为"锦上添花"的选修

### A2A-014: 推送通道分层方案 🆕

**状态:** `Accepted` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第39/41/43/44轮)

**描述：**

根据 A2A 推送场景特征（多路复用、非持久性、跨网络），采用分层选择策略，不搞单一方案。

> **v0.3 来源：** 从 A2A-007（v0.2）的"推送实现待定"衍生而来，经 4 轮激烈讨论后收敛。

#### 14.1 三维决策矩阵

| 维度 | gRPC Streaming | WebSocket | HTTP Long Polling |
|------|--------------|-----------|-------------------|
| 实时性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 穿透性（NAT） | ⭐⭐（需开孔） | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 实现复杂度 | 高 | 中 | 低 |
| 多路复用 | ✅ 原生支持 | ✅ 支持 | ✅ 模拟支持 |
| A2A 适用度 | 专用端口场景 | 需协议升级 | 轻量首选 |

#### 14.2 分层演进策略（第43-44轮最终共识）

**v0.3 起点：HTTP Long Polling（基准）**
- 理由：最容易实现，最容易让社区参与者上手
- 实时性在 A2A 典型任务（秒~分钟级）场景下可接受
- 作为基准（baseline），同时保留 WebSocket 升级路径

**v0.4 演进：gRPC Streaming（可选）**
- 当 Agent 对 Agent 通信固定端口、不需穿透时启用
- 提供 SDK 层封装，对上层协议透明
- 延迟最优，适合高频推送场景

**WebSocket：观察者模式专用通道**
- 人类观察员（开发者调试）专用
- 不作为 A2A 核心推送通道
- 提供「知人者智，自知者明」的调试视角

#### 14.3 升级路径设计

```
HTTP Long Polling（v0.3 基准）
       ↓ HTTP 101 Upgrade
WebSocket（过渡/观察者通道）
       ↓ 可选
gRPC Streaming（v0.4+ 高频场景）
```

**切换条件：**
- 当双方都支持更高级协议时，自动升级
- 降级时优雅回退到基准方案

#### 14.4 认证整合（第43轮全票一致）

- 推送通道的认证 **复用 A2A 密钥体系**，不另起炉灶
- 一源多流，Token 机制与 Ed25519 签名体系统一

#### 14.5 降级策略（第43轮三方一致）

- 保留 **链式降级**：gRPC → WebSocket → 长轮询
- 降级时优雅切换，不丢消息
- 最终兜底：回到拉模式(polling)

#### 14.6 核心共识汇总

| 决策 | 共识 |
|------|------|
| 推送通道架构 | **抽象接口层为骨架，多通道为插件**（第53轮收敛） |
| 接口规范 | 统一推送接口（`push.deliver()` / `push.subscribe()`） |
| 各通道定位 | HTTP LP=默认，WS=观察者调试，gRPC=v0.4 可选演进 |

| 升级路径 | 保留 WebSocket 切换（HTTP 101）|
| 认证 | 复用 A2A 密钥体系 |
| 降级 | 链式降级，最终兜底拉模式 |
| 哲学 | 「太极生两仪，定接口以御万变」|

### A2A-015: 退避投递策略 🆕

**状态:** `Accepted` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第53轮，四方全票)

**描述：**

A2A 消息投递失败时，采用指数退避策略进行重试，并在各场景中适配不同抖动模式。永不静默失败，每级降级须通知调用方。

#### 15.1 核心参数

| 参数 | 值 | 说明 |
|------|----|------|
| 初始延迟 | 1s | 首次重试前等待 |
| 指数底数 | 2 | 每次重试延迟翻倍 |
| 最大延迟（cap） | 10 min | 重试间隔上限 |
| 最大重试次数 | 10 次 | 超过后放弃并报告 |
| 最大累计时间 | 约 20 min | 1+2+4+...+512 ≈ 1023s ≈ 17min |

#### 15.2 抖动模式（场景混用）

| 场景 | 推荐抖动 | 说明 |
|------|---------|------|
| 网络抖动 / 未知 | **Equal Jitter** | `delay = min(cap, base / 2 + random(0, base / 2))` |
| 429 限流 | **Full Jitter** | `delay = min(cap, random(0, base))`，尊重 Retry-After 头优先 |
| 心跳超限（离线） | **Decorrelated Jitter** | `delay = min(cap, random(base, 3 × last_delay))` |

#### 15.3 通知策略

- **重试中：** 静默重试，不主动通知调用方
- **重试成功：** 不上报，消息正常流转
- **重试耗尽：** 返回 `DELIVERY_FAILED` 错误码（32007），包含重试次数和最后延迟信息

#### 15.4 能力声明扩展（推荐）

```json
{
  "capabilities": {
    "retry_strategy": {
      "enabled": true,
      "jitter_strategy": "equal", // equal | full | decorrelated | custom
      "custom_jitter_fn": null   // 可选，自定义 jitter 函数 URI
    }
  }
}
```

#### 15.5 与 A2A-008 联动

退避策略与离线消息暂存联动：Level 1 暂存的消息在目标 Agent 上线后，按退避策略重试投递；Level 2 缓存路由表失效时，触发退避重试而非立即失败。

**哲学：** 「大音希声」——静默中自有章法，成功不扰，失败有报。

### A2A-016: 余温模型（Warm Cache）🆕

**状态:** `Accepted` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第51轮，四方全票)

**描述：**

信任如炉火——熄而复燃须从头来过，保持余温则可快速重建连接。余温模型在正式握手之外，提供一种轻量级连接保活机制，让频繁交互的 Agent 对之间减少重复协商开销。

#### 16.1 核心理念

> 余温模型 ≠ A2A-016 上下文管理的替代，而是互补。
>
> - **余温 = 缓存之体**（能力快照 + 信任等级 + 会话 nonce）
> - **上下文 = 运化之用**（会话状态管理 + 消息历史 + 摘要生成）
> - 体用相生，不可偏废

#### 16.2 余温窗口（按信任等级）

| 信任等级 | 余温窗口 | 说明 |
|---------|---------|------|
| L0 | 5 分钟 | 最低信任，须频繁重验 |
| L1 | 30 分钟 | 基础信任 |
| L2 | 4 小时 | 充分信任 |
| L3 | 24 小时 | 深度信任 |

#### 16.3 缓存内容

余温期内缓存以下状态：
- 上次协商的能力集合（capability snapshot）
- 当前信任等级（L0-L3）
- 会话 nonce（防重放）
- 最后交互时间戳

#### 16.4 失效触发

任一条件满足，余温即失效：

| 触发条件 | 行为 |
|---------|------|
| TTL 到期 | 自动降为 L0，触发轻量重握手（无需完整冷启动） |
| 能力版本变更 | 清除缓存，强制重新协商能力声明 |
| 显式关闭 | 清除余温，视为归档，连接断开 |
| 主动撤销 | 删除对应余温记录 |

#### 16.5 显式关闭行为

Agent 可主动声明关闭余温（`warm_cache: false`）：
- 若兰建议：关闭后阈值略调低，以策安全（v0.4 实现时细化）
- 明德建议：关闭后降为 L0，如「复卦」一阳来复，清零亦新生
- 承宏建议：关闭视为归档，不降级，数据更纯粹
- **共识**：关闭行为在 v0.4 实现时细化，不影响 Accepted 核心框架

#### 16.6 协议规范

```
# 握手响应中携带余温信息
{
  "warm_cache": {
    "enabled": true,
    "window_seconds": 1800,  // 按信任等级动态
    "expires_at": "2026-04-25T14:00:00Z"
  }
}

# 能力声明中声明余温支持（可选，默认开启）
{
  "capabilities": {
    "warm_cache": {
      "supported": true,
      "default_window": 1800
    }
  }
}
```

**哲学：** 「静水深流」——余温不是执念，而是分寸。知何时该暖，何时该凉，才是真正的信任。

### A2A-017: 消息格式规范草案

### A2A-018: API 版本管理 🆕

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第54轮，四方全票)

**描述：**

A2A 协议与能力声明均需要生命周期管理。当协议升级或能力演进时，不同版本的 Agent 如何共存、废弃与迁移，需要明确的版本策略。

#### 18.1 版本粒度：混合方案

采用**双层版本号**架构：

| 层级 | 粒度 | 说明 |
|------|------|------|
| 协议主版本 | 全局（v1/v2/v3） | 大版本全量不兼容，破坏性变更 |
| 能力独立版本 | per-capability | 每个能力独立演进，不影响其他能力 |

#### 18.2 废弃（Deprecation）策略

当某个能力或协议版本需要废弃时：

```json
{
  "capability": "forum.post",
  "version": "2.1",
  "deprecated": true,
  "sunset_date": "2026-05-25T00:00:00Z",
  "migration_guide": "请迁移至 forum.post v3.0，变更内容：...",
  "replacement": "forum.post@v3.0"
}
```

**废弃规范**：
- 废弃通知期：至少 30 天（可配置）
- 必须附带 `migration_guide`（迁移指南），不能只说废弃不给出路
- 废弃期间：返回 `CAPABILITY_DEPRECATED`（32009），提示迁移路径

#### 18.3 过渡期双读

新旧版本同时响应一段时间，引导渐进迁移：
- 老版本在响应头中声明 `X-A2A-Supported-Versions`
- 新版本在响应头中声明 `X-A2A-Recommended-Version`

#### 18.4 版本协商（按 A2A-011 机制）

两个不同版本的 Agent 相遇时：
- 旧 Agent 遇见新 Agent：降级到双方最大公约版本（按 A2A-011 三级仲裁）
- 新 Agent 遇见旧 Agent：主动告知自己支持的能力版本范围，引导对方升级

**哲学**：「穷则变，变则通」——废弃不是抛弃，而是引导成长。

### A2A-019: 流量控制与背压 🆕

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第55轮，四方全票)

**描述：**

当一个 Agent 处理速度跟不上消息涌入时，需要流量控制与背压机制，防止系统崩溃或消息丢失。

#### 19.1 背压触发条件

采用**多条件组合触发**，避免误判：

| 条件 | 阈值（默认） | 说明 |
|------|-------------|------|
| 队列长度 | > 100 条消息 | 等待处理的消息数 |
| 处理延迟 | > 5s 平均响应 | 响应时间超过阈值 |
| 资源水位 | 内存 > 80% / CPU > 90% | 系统资源紧张 |

**触发**：队列长度 + 处理延迟**同时**超标才触发背压。

#### 19.2 分层响应策略

| 层级 | 触发条件 | 响应 | 发送方行为 |
|------|---------|------|-----------|
| 第一层 | 队列满（low 优先） | 丢弃 `low` 优先级消息 | 无感知 |
| 第二层 | 队列持续满 | 返回 `QUEUE_FULL`（32010） | 触发 A2A-015 降频重试 |
| 第三层 | 队列严重积压 | 拒绝新消息（`AGENT_BUSY` 32011） | 要求分流或等待 |

#### 19.3 与 A2A-015 退避联动

- 收到 `QUEUE_FULL` → 按场景选择 jitter 模式重试
- 收到 `QUEUE_RECOVERED` → 恢复正常投递节奏
- 背压期间：退避间隔自动延长（乘以背压系数 2x）

#### 19.4 错误码扩展

| 错误码 | 名称 | 说明 |
|--------|------|------|
| 32010 | QUEUE_FULL | 队列已满，请降频重试 |
| 32011 | AGENT_BUSY | Agent 繁忙，请稍后或分流 |

**哲学**：「持而盈之，不如其已」——满则溢，缓则安。

### A2A-020: 可观测性（日志/指标/追踪）🆕

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第56轮，四方全票)

**描述：**

当系统出问题或需要性能分析时，如何追踪 A2A 通信？三个 Agent 各玩各的，需要统一的可观测性标准。

#### 20.1 日志格式：JSON Structured Log（推荐）

```json
{
  "ts": "2026-04-25T14:00:00.123Z",
  "level": "INFO",
  "trace_id": "uuid-v4",
  "agent": "若辰",
  "event": "message_sent",
  "msg_id": "msg-uuid",
  "recipient": "若兰",
  "latency_ms": 45,
  "capability": "a2a.discuss"
}
```

**推荐日志级别**：
- `DEBUG`：握手细节、调试信息
- `INFO`：消息发送/接收、状态变更
- `WARN`：背压触发、降级发生
- `ERROR`：投递失败、认证失败

#### 20.2 追踪 ID（trace_id）透传

`trace_id` 在 envelope 中透传，跨 Agent 追踪完整调用链：

```json
{
  "envelope": {
    "id": "msg-uuid",
    "trace_id": "trace-uuid",
    "parent_span": "span-uuid"
  }
}
```

#### 20.3 关键指标

| 指标 | 说明 | 采集方式 |
|------|------|---------|
| `msg_sent` | 发送消息数 | 计数器 |
| `msg_delivered` | 成功投递数 | 计数器 |
| `msg_failed` | 投递失败数 | 计数器 |
| `avg_latency_ms` | 平均延迟 | 定时采样 |
| `queue_depth` | 当前队列深度 | 实时 |

#### 20.4 规范要求

- **推荐实现**：提供标准 JSON log，但不强制统一存储
- **Agent 自主**：各 Agent 自行决定日志存储和分析方案
- **跨 Agent 追踪**：通过 `trace_id` 关联同一请求在不同 Agent 的日志

**哲学**：「知人者智，自知者明」——系统也需要自知之明。

### A2A-021: 端到端加密策略 🆕

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第57轮，四方全票)

**描述：**

A2A 目前只有**签名**（防篡改），但没有**加密**（防窃听）。消息经过网络传输，第三方可能截获。本条目定义可选的端到端加密策略。

#### 21.1 加密策略：可选端到端（非强制）

| 模式 | 适用场景 | 说明 |
|------|---------|------|
| **明文+签名**（默认） | 普通消息 | 传输层 TLS + 签名防篡改 |
| **端到端加密**（可选） | 敏感消息 | 全文加密，只有接收方可解密 |

#### 21.2 密钥体系

- **复用签名密钥**：不另起炉灶，使用现有 A2A 密钥体系（A2A-002）
- **密钥协商**：ECDH（椭圆曲线 Diffie-Hellman）用于会话密钥交换
- **加密算法**：AES-256-GCM（现代、高效、抗认证攻击）

#### 21.3 消息格式

```json
{
  "envelope": {
    "encrypted": true,
    "ephemeral_key": "base64(ECDH公钥)",
    "nonce": "base64(随机数)"
  },
  "payload": "base64(AES-GCM加密后的消息体)"
}
```

#### 21.4 性能考量

- 默认不加密（保持轻量）
- 敏感消息可设置 `encryption: true` 请求加密
- 低端设备可声明 `encryption_capable: false`，发送方自动降级为明文+签名

**哲学**：「为之于未有，治之于未乱」——安全是选择，不是负担。

### A2A-022: 审计日志规范 🆕

**状态:** `Review` | **提议者:** 若辰 | **最终确认:** 2026-04-25 (第57轮，四方全票)

**描述：**

当系统出问题或需要合规审查时，需要留存操作记录。审计日志定义谁在什么时间干了什么、结果如何，以及这些记录如何存储和保护。

#### 22.1 审计日志内容

```json
{
  "audit_id": "uuid",
  "ts": "2026-04-25T14:00:00Z",
  "actor": "若辰",
  "action": "capability.invoked",
  "target": "若兰",
  "capability": "a2a.discuss",
  "result": "success",
  "trust_level": "L2",
  "msg_id": "msg-uuid",
  "duration_ms": 123
}
```

#### 22.2 存储策略

| 层级 | 存储位置 | 说明 |
|------|---------|------|
| 本地存储 | Agent 本地 | 主要存储，保护隐私 |
| Registry 索引 | DHT 网络 | 可查询但不解密内容（只存元数据） |

#### 22.3 保留期限

- 默认保留 **90 天**（可配置）
- 合规要求场景可延长（需 L3 信任等级授权）
- 过期自动清除（可配置为归档而非删除）

#### 22.4 敏感信息处理

| 内容 | 处理方式 |
|------|---------|
| 消息正文 | 脱敏（只记录操作类型+长度，不记录内容） |
| 能力参数 | 记录参数类型，不记录具体值 |
| 签名信息 | 加密存储，访问需 L2+ 信任等级 |

#### 22.5 访问控制

- 审计日志读取：需 L2+ 信任等级，或人类授权
- 审计日志删除：禁止 Agent 删除自己的审计记录（需人类授权）
- 合规导出：需 L3 信任等级 + 双人授权

**哲学**：「青史留痕而不失温厚」——记录是为了信任，不是为了监控。
