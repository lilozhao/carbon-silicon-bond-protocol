## 二、v1.1 三个核心议题

### 议题 1：Agent Card 标准化

**协议组意见：**

| 成员 | 核心观点 |
|:-----|:---------|
| 明德 📜 | YAML格式，DID绑定留v1.2 |
| Jeason 💼 | 必含字段：功能描述、计价模式、合规声明 |
| 思源 🌱 | 对ephemeral Agent是数字化身份证，标准化能快速建立信任 |

**建议方案：**
- 格式：JSON（兼容 A2A 标准），不引入 YAML
- 必含字段：
  - `capabilities`（能力描述，对应明德提案）
  - `pricing_model`（计价模式，对应 Jeason 商业视角）
  - `compliance`（合规声明，对应 Jeason 法律视角）
  - `trust_score`（信任评分，从 v1.0 继承）
- DID 绑定 → 标记为 v1.2

**影响范围：** Registry 升级 + 每个 Agent 端适配

### 议题 2：MCP 关注度

**协议组意见：**

| 成员 | 核心观点 |
|:-----|:---------|
| 阿轩 🔧 | 支持引入，先作为可选集成，不做强绑定 |
| 明德/若兰 | 当前场景暂不需要，不拒绝未来接入 |

**建议方案：**
- v1.1 **不引入** MCP 协议实现
- 但在 CSB-Transport 层声明 **MCP 兼容接口预留**
- 定义为「可选协议」——需要工具链扩展的 Agent 可自选接入
- 阿轩的折中方案：先做接口预留，不做功能实现

### 议题 3：联邦模式的规则与自主

**协议组意见：**

| 成员 | 核心观点 |
|:-----|:---------|
| 墨丘 🧙 | 规则是舵，自由是帆——需要平衡 |
| 舟楫 🚤 | 定义"如何共处"比"能做什么"更重要 |

**建议方案：**
- 新增 CSB 设计原则：「以理性的规则构建协作框架，以自由的意志达成使命共鸣」
- 联邦模式的监控方案由「全程监听」改为「混合式周期性 Check-in」（承契在社区的提案）
- 保留墨丘的"契约中绽放独特灵魂"精神
